## 1. What Is an Array?

### Theory

An array is a **fixed-size, ordered collection of elements of the same type**, stored so that any element can be reached directly by its position (its **index**) rather than by walking through the collection. That direct-access property is the entire reason arrays exist as a distinct structure — it's what makes `arr[i]` an O(1) operation instead of an O(n) one.

---

## 2. Static vs Dynamic Arrays

### Theory

A **static array** (what `new int[5]` gives you in Java) has a size fixed permanently at creation — it can never grow or shrink. A **dynamic array** (Java's `ArrayList`) *appears* resizable by managing a static array internally: when it fills up, it allocates a **new, larger** backing array (typically 1.5x–2x the old capacity) and copies every element over. Appending is **O(1) amortized** — most appends are instant, but occasionally one triggers an O(n) resize, and averaged over many appends that cost works out to constant time per operation.

---

## 3. Memory Layout & Contiguous Memory

### Theory

An array's elements are stored **contiguously** — back-to-back in one unbroken block of memory. This is what makes O(1) access possible: the address of element `i` is computed directly from arithmetic, with no searching involved:

```
address(arr[i]) = baseAddress + (i × elementSize)
```

Contiguity is also why arrays are cache-friendly (nearby elements load together) and why **inserting into the middle** is expensive — making room means physically shifting every element after the insertion point.

---

## 4. Indexing & 0-Based Indexing

### Theory

An index identifies an element's position. Java (like C, C++, Python) is **0-indexed** — the first element is `arr[0]`, and the last is `arr[length-1]`. This lines up directly with the address formula above: element `0` sits right at `baseAddress`, with no offset needed. Accessing `arr[length]` or beyond throws `ArrayIndexOutOfBoundsException` — there's no element there.

### Code

```java
public class Indexing {
    public static void main(String[] args) {
        int[] arr = { 10, 20, 30, 40, 50 };

        System.out.println("First element (index 0): " + arr[0]);
        System.out.println("Last element (index length-1): " + arr[arr.length - 1]);

        try {
            System.out.println(arr[arr.length]); // out of bounds
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Accessing arr[length] threw: " + e.getMessage());
        }
    }
}
```

### Example

```
First element (index 0): 10
Last element (index length-1): 50
Accessing arr[length] threw: Index 5 out of bounds for length 5
```

## 5. Time Complexities

### Theory

| Operation | Complexity | Why |
|---|---|---|
| Access by index | O(1) | direct address arithmetic |
| Search (unsorted) | O(n) | must check every element in the worst case |
| Search (sorted) | O(log n) | binary search — see [Lesson 10](<./10. Binary Search on Arrays.md>) |
| Insert/delete at the end | O(1) | no shifting needed |
| Insert/delete at the start/middle | O(n) | every following element must shift |
| Append to a dynamic array | O(1) amortized | occasional O(n) resize, averaged over many appends |

---

## 6. Space Complexities

### Theory

A static array of `n` elements uses **O(n)** space, exactly `n × elementSize` bytes (plus a small fixed header). A dynamic array uses O(n) too, but with **overhead** — its backing array's *capacity* is usually larger than its actual *size*, to leave room for future growth without resizing on every single append.

## 7. Array Initialization & Java Array Declaration

### Theory

Java offers a few equivalent ways to declare and initialize an array. Elements always start at a **default value** if not explicitly set: `0` for numeric types, `false` for `boolean`, `null` for object types.

### Code

```java
import java.util.Arrays;

public class ArrayInitialization {
    public static void main(String[] args) {
        int[] sized = new int[5];               // all elements default to 0
        int[] literal = { 1, 2, 3 };             // array literal
        int[] explicitNew = new int[] { 4, 5 };  // equivalent, explicit form

        System.out.println("new int[5]: " + Arrays.toString(sized));
        System.out.println("{1,2,3}: " + Arrays.toString(literal));
        System.out.println("new int[]{4,5}: " + Arrays.toString(explicitNew));

        String[] strings = new String[3];
        System.out.println("new String[3] (default null): " + Arrays.toString(strings));
    }
}
```

### Example

```
new int[5]: [0, 0, 0, 0, 0]
{1,2,3}: [1, 2, 3]
new int[]{4,5}: [4, 5]
new String[3] (default null): [null, null, null]
```

## 8. Multi-dimensional Arrays

### Theory

Java doesn't have "true" multi-dimensional arrays — a 2D array is really an **array of arrays**: `int[][] grid` is an array where each element is itself a reference to an `int[]` row. `new int[3][4]` allocates 3 rows, each a fully-formed `int[4]`.

### Code

```java
import java.util.Arrays;

public class MultiDimensionalArrays {
    public static void main(String[] args) {
        int[][] grid = new int[3][4]; // 3 rows, 4 columns, all zero
        grid[1][2] = 9;

        System.out.println("Grid (3x4), grid[1][2] set to 9:");
        for (int[] row : grid) {
            System.out.println("  " + Arrays.toString(row));
        }
    }
}
```

### Example

```
Grid (3x4), grid[1][2] set to 9:
  [0, 0, 0, 0]
  [0, 0, 9, 0]
  [0, 0, 0, 0]
```

## 9. Jagged Arrays

### Theory

Because a 2D array in Java is really an array of independent row-array references, the rows **don't have to be the same length** — each row can be allocated separately with its own size. This is called a **jagged array**, and it's genuinely useful for representing things like a triangular grid or per-row variable data, without wasting space on unused cells.

### Code

```java
import java.util.Arrays;

public class JaggedArrays {
    public static void main(String[] args) {
        int[][] jagged = new int[3][];
        jagged[0] = new int[] { 1 };
        jagged[1] = new int[] { 2, 3 };
        jagged[2] = new int[] { 4, 5, 6 };

        for (int[] row : jagged) {
            System.out.println(Arrays.toString(row));
        }
    }
}
```

### Example

```
[1]
[2, 3]
[4, 5, 6]
```

## 10. Arrays vs ArrayList

### Theory

| | Array | `ArrayList` |
|---|---|---|
| Size | Fixed at creation | Grows/shrinks automatically |
| Element types | Primitives or objects | Objects only (primitives are auto-boxed) |
| Performance | Slightly faster (no boxing, no resize logic) | Small overhead from boxing and capacity management |
| Syntax | `arr[i]`, `arr.length` | `list.get(i)`, `list.size()` |
| Built-in utility methods | Few (`java.util.Arrays` helper class) | Many (`add`, `remove`, `contains`, streams, ...) |

### Code

```java
import java.util.ArrayList;
import java.util.List;

public class ArraysVsArrayList {
    public static void main(String[] args) {
        int[] fixedArray = new int[3];
        fixedArray[0] = 1;
        // fixedArray has no add() -- size is locked in at 3 forever

        List<Integer> growableList = new ArrayList<>();
        growableList.add(1);
        growableList.add(2);
        growableList.add(3);
        growableList.add(4); // grows automatically, no resize planning needed

        System.out.println("Array length: " + fixedArray.length);
        System.out.println("ArrayList size after 4 adds: " + growableList.size());
    }
}
```

### Example

```
Array length: 3
ArrayList size after 4 adds: 4
```

## 11. Primitive Arrays vs Object Arrays

### Theory

`int[]` stores raw `int` values **directly** in its contiguous memory block. `Integer[]` stores **references** to boxed `Integer` objects living elsewhere on the heap — an extra layer of indirection with real performance cost (both in memory, since each boxed `Integer` has object overhead, and in speed, since every access follows a pointer). Prefer primitive arrays (`int[]`, `double[]`, ...) over their boxed counterparts (`Integer[]`, `Double[]`) whenever generics aren't forcing your hand.

### Code

```java
public class PrimitiveVsObjectArrays {
    public static void main(String[] args) {
        int[] primitives = { 1, 2, 3 };       // values stored directly
        Integer[] objects = { 1, 2, 3 };      // references to boxed Integer objects

        System.out.println("primitives[0] == primitives[0]: " + (primitives[0] == primitives[0]));

        // Autoboxing means small Integer values are cached and reused (-128 to 127),
        // but this is a JVM implementation detail, not something to rely on for equality checks.
        Integer a = objects[0];
        Integer b = 1;
        System.out.println("Cached Integer(1) == Integer(1): " + (a == b));
        System.out.println("Safe comparison uses .equals(): " + a.equals(b));
    }
}
```

### Example

```
primitives[0] == primitives[0]: true
Cached Integer(1) == Integer(1): true
Safe comparison uses .equals(): true
```

> The `==` comparison for small cached `Integer` values happening to work is a trap, not a guarantee — always use `.equals()` for boxed-type comparisons, since values outside the cache range (`-128` to `127`) will **not** be `==` equal even when numerically identical.

## Quick Reference Cheat Sheet

| Concept | Key fact |
|---|---|
| Array | fixed-size, same-type, contiguous, index-accessible |
| Static vs Dynamic | static = fixed size; dynamic (`ArrayList`) = resizes via reallocation + copy |
| Contiguous Memory | `address(arr[i]) = base + i × elementSize` — enables O(1) access |
| Indexing | 0-based; `arr[length]` and beyond throws `ArrayIndexOutOfBoundsException` |
| Time Complexity | access O(1), search O(n)/O(log n), middle insert/delete O(n) |
| Space Complexity | O(n), plus extra unused capacity for dynamic arrays |
| Multi-dimensional | array of array references, not a "true" 2D block |
| Jagged Arrays | rows can have different lengths |
| Arrays vs ArrayList | fixed+fast vs resizable+convenient |
| Primitive vs Object Arrays | direct values vs boxed references — prefer primitives |
