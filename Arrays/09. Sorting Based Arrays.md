# 9. Sorting Based Arrays

Sorting is often the first move that makes an array problem tractable — many two-pointer, greedy, and interval techniques only work *because* the input is sorted first. This lesson covers what "sorted" actually guarantees (stability, in-place-ness), how to customize the order, and the non-comparison sorts that beat O(n log n) when the data has special structure.

## 1. Sorting

### Theory

Sorting arranges elements into a defined order (usually ascending). It's covered from the API side in [Lesson 3: Arrays.sort()](<./03. Java Array APIs.md#1-arrayssort>) — this lesson focuses on the **properties** a sort can have (stable? in-place?) and the specialized non-comparison sorts, rather than re-deriving comparison-based sorting algorithms from scratch.

---

## 2. Stable Sorting

### Theory

A sort is **stable** if elements that compare as *equal* keep their **original relative order** after sorting. This matters whenever you sort by one key but care about a secondary, implicit order — e.g. sorting a list of records by score while keeping same-score records in their original (perhaps insertion or alphabetical) order. Java's `Arrays.sort()` on **object arrays** (and `Collections.sort()`) uses a stable Timsort; on **primitive arrays** it isn't (and doesn't need to be — identical primitive values are indistinguishable, so "order" between equal `int`s is meaningless).

### Code

```java
import java.util.ArrayList;
import java.util.Arrays;
import java.util.List;

public class StableSorting {
    record Entry(int key, String label) {}

    public static void main(String[] args) {
        List<Entry> data = new ArrayList<>(List.of(
                new Entry(3, "a"), new Entry(1, "b"), new Entry(3, "c"),
                new Entry(2, "d"), new Entry(1, "e")
        ));

        data.sort((x, y) -> Integer.compare(x.key(), y.key()));
        System.out.println("Stable sort by key: " + data);
    }
}
```

### Example

```
Stable sort by key: [Entry[key=1, label=b], Entry[key=1, label=e], Entry[key=2, label=d], Entry[key=3, label=a], Entry[key=3, label=c]]
```

Both `key=1` entries keep `b` before `e` (their original order), and both `key=3` entries keep `a` before `c` — that's stability.

## 3. In-place Sorting

### Theory

An **in-place** sort rearranges elements using only **O(1)** extra memory (or O(log n) for recursion stack space), rather than allocating a second array. Quicksort and heapsort are in-place; standard merge sort is not (it needs an auxiliary array to merge into). This distinction matters when memory is constrained or the array is huge.

| Algorithm | In-place? | Stable? |
|---|---|---|
| Quicksort | Yes | No |
| Heapsort | Yes | No |
| Merge Sort | No (needs O(n) extra space) | Yes |
| Insertion Sort | Yes | Yes |

## 4. Custom Comparator

### Theory

Sorting objects (or applying a non-default order to primitives boxed as objects) needs a `Comparator` describing how to compare two elements. This is essential whenever "ascending numeric order" isn't the goal — sorting strings by length, sorting intervals by start time, sorting a 2D array by a specific column.

### Code

```java
import java.util.Arrays;
import java.util.Comparator;

public class CustomComparator {
    public static void main(String[] args) {
        String[] words = { "banana", "kiwi", "apple", "fig" };

        Arrays.sort(words, Comparator.comparingInt(String::length));
        System.out.println("Sorted by length: " + Arrays.toString(words));

        int[][] intervals = { { 3, 4 }, { 1, 5 }, { 2, 2 } };
        Arrays.sort(intervals, Comparator.comparingInt(interval -> interval[0]));
        System.out.println("Sorted by start value: " + Arrays.deepToString(intervals));
    }
}
```

### Example

```
Sorted by length: [kiwi, fig, banana, apple]
Sorted by start value: [[1, 5], [2, 2], [3, 4]]
```

> `Arrays.sort` with a `Comparator` requires an **object** array (`String[]`, `int[][]`, etc.) — primitive arrays like `int[]` can't take a `Comparator` at all, since Java generics don't work on primitives directly.

## 5. Partial Sorting

### Theory

Sometimes only **part** of the sorted order is actually needed — the top `k` largest elements, or just the `k`-th smallest — and fully sorting the whole array (O(n log n)) does more work than necessary. A **heap of size `k`** (full coverage in [Lesson 16: Heap + Arrays](<./16. Heap + Arrays.md>)) finds the top `k` in O(n log k), better than sorting when `k` is small relative to `n`.

## 6. Counting Sort Idea

### Theory

**Counting Sort** is a non-comparison sort for integers in a small, known range `[0, maxValue]`: count how many times each value occurs (see [Lesson 4: Frequency Problems](<./04. Frequency Problems.md#3-array-frequency>)), then reconstruct the sorted array by writing each value out as many times as it occurred. Runs in **O(n + maxValue)** — beats O(n log n) comparison sorts whenever `maxValue` isn't too large.

### Code

```java
import java.util.Arrays;

public class CountingSortIdea {
    public static void main(String[] args) {
        int[] arr = { 4, 2, 2, 8, 3, 3, 1 };
        System.out.println("Counting sort: " + Arrays.toString(countingSort(arr, 8)));
    }

    static int[] countingSort(int[] arr, int maxValue) {
        int[] count = new int[maxValue + 1];
        for (int value : arr) count[value]++;

        int[] result = new int[arr.length];
        int index = 0;
        for (int value = 0; value <= maxValue; value++) {
            for (int i = 0; i < count[value]; i++) {
                result[index++] = value;
            }
        }
        return result;
    }
}
```

### Example

```
Counting sort: [1, 2, 2, 3, 3, 4, 8]
```

## 7. Bucket Idea

### Theory

**Bucket Sort** generalizes counting sort to values that aren't small integers (e.g. floats spread across a range): divide the value range into `k` equal-width buckets, drop each element into its bucket, sort each bucket individually (with any simple sort, since buckets are small), then concatenate the buckets in order. Works best when values are roughly **uniformly distributed** across the range — clustered data defeats the purpose, since one bucket would end up doing almost all the work.

### Code

```java
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class BucketIdea {
    public static void main(String[] args) {
        double[] arr = { 0.42, 0.32, 0.23, 0.52, 0.25, 0.75 };
        System.out.println("Bucket sort: " + java.util.Arrays.toString(bucketSort(arr, 5)));
    }

    static double[] bucketSort(double[] arr, int bucketCount) {
        double min = Double.MAX_VALUE, max = -Double.MAX_VALUE;
        for (double value : arr) {
            min = Math.min(min, value);
            max = Math.max(max, value);
        }

        List<List<Double>> buckets = new ArrayList<>();
        for (int i = 0; i < bucketCount; i++) buckets.add(new ArrayList<>());

        double rangeSize = (max - min) / bucketCount;
        for (double value : arr) {
            int bucketIndex = (int) ((value - min) / rangeSize);
            if (bucketIndex == bucketCount) bucketIndex--; // the max value lands in the last bucket
            buckets.get(bucketIndex).add(value);
        }

        double[] result = new double[arr.length];
        int index = 0;
        for (List<Double> bucket : buckets) {
            Collections.sort(bucket);
            for (double value : bucket) result[index++] = value;
        }
        return result;
    }
}
```

### Example

```
Bucket sort: [0.23, 0.25, 0.32, 0.42, 0.52, 0.75]
```

## Quick Reference Cheat Sheet

| Concept | Key fact |
|---|---|
| Stable Sorting | equal elements keep their original relative order |
| In-place Sorting | O(1) extra space (quicksort, heapsort — not standard merge sort) |
| Custom Comparator | required for non-default order; only works on object arrays |
| Partial Sorting | heap of size `k` beats full sort when only top-`k` is needed |
| Counting Sort | O(n + maxValue), for small-range integers |
| Bucket Sort | divide into buckets, sort each, concatenate — best for uniform data |
