# 9. Sorting Based Arrays

Sorting is often the first move that makes an array problem tractable — many two-pointer, greedy, and interval techniques only work *because* the input is sorted first. This lesson goes from the simplest O(n²) sorts through the O(n log n) comparison sorts to the non-comparison sorts that beat O(n log n) when the data has special structure, then wraps up with the properties (stability, in-place-ness, custom ordering) that distinguish them.

## 1. Sorting Overview

### Theory

Every sorting algorithm answers the same question — "put these elements in order" — with a different trade-off between time complexity, space complexity, and stability. Here's the full landscape covered in this lesson, to orient before diving into each:

| Algorithm | Best | Average | Worst | Space | Stable? | In-place? |
|---|---|---|---|---|---|---|
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) | Yes | Yes |
| Selection Sort | O(n²) | O(n²) | O(n²) | O(1) | No | Yes |
| Insertion Sort | O(n) | O(n²) | O(n²) | O(1) | Yes | Yes |
| Merge Sort | O(n log n) | O(n log n) | O(n log n) | O(n) | Yes | No |
| Quick Sort | O(n log n) | O(n log n) | O(n²) | O(log n) | No | Yes |
| Heap Sort | O(n log n) | O(n log n) | O(n log n) | O(1) | No | Yes |
| Counting Sort | O(n + k) | O(n + k) | O(n + k) | O(k) | Yes | No |
| Radix Sort | O(d(n + k)) | O(d(n + k)) | O(d(n + k)) | O(n + k) | Yes | No |
| Bucket Sort | O(n + k) | O(n + k) | O(n²) | O(n) | Yes | No |
| Shell Sort | O(n log n) | O(n^1.25) | O(n²) | O(1) | No | Yes |

`k` = range of input values, `d` = number of digits. Java's `Arrays.sort()` (covered in [Lesson 3](<./03. Java Array APIs.md#1-arrayssort>)) uses Dual-Pivot Quicksort for primitives and Timsort (a hybrid of merge sort and insertion sort) for objects — you rarely hand-roll these in practice, but knowing how each works is what makes it possible to reason about *when* a non-default sort (counting/radix/bucket) would actually be faster.

## 2. Bubble Sort

### Theory

Repeatedly walk the array, swapping every **adjacent** out-of-order pair. Each full pass "bubbles" the largest unsorted element to its correct position at the end. Simple but slow — mainly useful for teaching, or arrays that are already nearly sorted (where the early-exit optimization below makes it closer to O(n)).

### Code

```java
import java.util.Arrays;

public class BubbleSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        bubbleSort(arr);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void bubbleSort(int[] arr) {
        int n = arr.length;
        for (int i = 0; i < n; i++) {
            boolean swapped = false;
            for (int j = 0; j < n - 1 - i; j++) {
                if (arr[j] > arr[j + 1]) {
                    int temp = arr[j];
                    arr[j] = arr[j + 1];
                    arr[j + 1] = temp;
                    swapped = true;
                }
            }
            if (!swapped) break; // already sorted -- no swaps happened this pass
        }
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 3. Selection Sort

### Theory

For each position, find the **minimum** of the remaining unsorted elements and swap it into place. Unlike bubble sort, this makes at most `n` swaps total (one per position) — useful when swaps are expensive (e.g. large records), even though comparisons are still O(n²).

### Code

```java
import java.util.Arrays;

public class SelectionSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        selectionSort(arr);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void selectionSort(int[] arr) {
        int n = arr.length;
        for (int i = 0; i < n; i++) {
            int minIndex = i;
            for (int j = i + 1; j < n; j++) {
                if (arr[j] < arr[minIndex]) minIndex = j;
            }
            int temp = arr[i];
            arr[i] = arr[minIndex];
            arr[minIndex] = temp;
        }
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 4. Insertion Sort

### Theory

Build the sorted portion one element at a time: take the next unsorted element and shift it backward through the already-sorted prefix until it lands in its correct spot. Excellent for small or nearly-sorted arrays (Java's Timsort actually falls back to insertion sort for small runs).

### Code

```java
import java.util.Arrays;

public class InsertionSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        insertionSort(arr);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void insertionSort(int[] arr) {
        for (int i = 1; i < arr.length; i++) {
            int key = arr[i];
            int j = i - 1;
            while (j >= 0 && arr[j] > key) {
                arr[j + 1] = arr[j];
                j--;
            }
            arr[j + 1] = key;
        }
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 5. Merge Sort

### Theory

Divide the array in half recursively until each piece has one element (trivially sorted), then merge sorted halves back together (the same [Merge operation](<./02. Basic Operations.md#10-merge>) from Lesson 2). Guarantees **O(n log n)** in every case — no worst-case degradation — at the cost of O(n) extra space for the merge step.

### Code

```java
import java.util.Arrays;

public class MergeSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        int[] sorted = mergeSort(arr);
        System.out.println("Sorted: " + Arrays.toString(sorted));
    }

    static int[] mergeSort(int[] arr) {
        if (arr.length <= 1) return arr;

        int mid = arr.length / 2;
        int[] left = mergeSort(Arrays.copyOfRange(arr, 0, mid));
        int[] right = mergeSort(Arrays.copyOfRange(arr, mid, arr.length));

        return merge(left, right);
    }

    static int[] merge(int[] left, int[] right) {
        int[] result = new int[left.length + right.length];
        int i = 0, j = 0, k = 0;
        while (i < left.length && j < right.length) {
            result[k++] = (left[i] <= right[j]) ? left[i++] : right[j++];
        }
        while (i < left.length) result[k++] = left[i++];
        while (j < right.length) result[k++] = right[j++];
        return result;
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 6. Quick Sort

### Theory

Pick a **pivot**, partition the array so everything smaller ends up before it and everything larger ends up after (the same [Partitioning](<./08. Two Pointers.md#4-partitioning>) idea from Lesson 8), then recursively sort each side. Average **O(n log n)**, in-place — but worst case degrades to O(n²) on already-sorted input with a poorly chosen pivot (using the last element as pivot, as below, is the classic vulnerable case; randomized or median-of-three pivot selection avoids this in practice).

### Code

```java
import java.util.Arrays;

public class QuickSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        quickSort(arr, 0, arr.length - 1);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void quickSort(int[] arr, int lo, int hi) {
        if (lo >= hi) return;
        int pivotIndex = partition(arr, lo, hi);
        quickSort(arr, lo, pivotIndex - 1);
        quickSort(arr, pivotIndex + 1, hi);
    }

    static int partition(int[] arr, int lo, int hi) {
        int pivot = arr[hi];
        int i = lo;
        for (int j = lo; j < hi; j++) {
            if (arr[j] < pivot) {
                swap(arr, i, j);
                i++;
            }
        }
        swap(arr, i, hi);
        return i;
    }

    static void swap(int[] arr, int i, int j) {
        int temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 7. Heap Sort

### Theory

Build a **max-heap** out of the array (arrange it so every parent is `≥` its children), then repeatedly swap the root — the current maximum — to the end of the unsorted region and shrink that region by one, re-heapifying each time. In-place and guaranteed **O(n log n)**, but not stable (equal elements can be reordered by the heap operations).

### Code

```java
import java.util.Arrays;

public class HeapSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3 };
        heapSort(arr);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void heapSort(int[] arr) {
        int n = arr.length;
        for (int i = n / 2 - 1; i >= 0; i--) {
            heapify(arr, n, i);
        }
        for (int i = n - 1; i > 0; i--) {
            swap(arr, 0, i);
            heapify(arr, i, 0);
        }
    }

    static void heapify(int[] arr, int size, int root) {
        int largest = root;
        int left = 2 * root + 1, right = 2 * root + 2;

        if (left < size && arr[left] > arr[largest]) largest = left;
        if (right < size && arr[right] > arr[largest]) largest = right;

        if (largest != root) {
            swap(arr, root, largest);
            heapify(arr, size, largest);
        }
    }

    static void swap(int[] arr, int i, int j) {
        int temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
}
```

### Example

```
Sorted: [1, 2, 3, 5, 8, 9]
```

## 8. Counting Sort

### Theory

For integers in a small, known range `[0, maxValue]`, count how many times each value occurs (see [Lesson 4](<./04. Frequency Problems.md#3-array-frequency>)), then reconstruct the sorted output by writing each value out as many times as it occurred. **O(n + maxValue)** — beats every comparison sort above whenever `maxValue` isn't too large, since it never compares elements at all.

### Code

```java
import java.util.Arrays;

public class CountingSort {
    public static void main(String[] args) {
        int[] arr = { 4, 2, 2, 8, 3, 3, 1 };
        System.out.println("Sorted: " + Arrays.toString(countingSort(arr, 8)));
    }

    static int[] countingSort(int[] arr, int maxValue) {
        int[] count = new int[maxValue + 1];
        for (int value : arr) count[value]++;

        int[] result = new int[arr.length];
        int index = 0;
        for (int value = 0; value <= maxValue; value++) {
            for (int i = 0; i < count[value]; i++) {
                result[index++] = value;
            }
        }
        return result;
    }
}
```

### Example

```
Sorted: [1, 2, 2, 3, 3, 4, 8]
```

## 9. Radix Sort

### Theory

Sorts integers **digit by digit**, from least significant to most significant, using a stable counting-style pass (bucketing `0-9`) at each digit position. Since each digit pass is stable, sorting by increasingly significant digits progressively refines the order without undoing earlier work. Runs in **O(d × (n + k))** where `d` is the number of digits and `k=10` (the digit base) — effectively linear for fixed-width numbers.

### Code

```java
import java.util.Arrays;

public class RadixSort {
    public static void main(String[] args) {
        int[] arr = { 170, 45, 75, 90, 802, 24, 2, 66 };
        System.out.println("Sorted: " + Arrays.toString(radixSort(arr)));
    }

    static int[] radixSort(int[] arr) {
        int max = Arrays.stream(arr).max().getAsInt();
        int[] result = arr.clone();

        for (long exp = 1; max / exp > 0; exp *= 10) {
            result = countingSortByDigit(result, exp);
        }
        return result;
    }

    static int[] countingSortByDigit(int[] arr, long exp) {
        int n = arr.length;
        int[] output = new int[n];
        int[] count = new int[10];

        for (int value : arr) count[(int) ((value / exp) % 10)]++;
        for (int i = 1; i < 10; i++) count[i] += count[i - 1]; // prefix sum -> positions

        for (int i = n - 1; i >= 0; i--) { // iterate backward to keep the sort stable
            int digit = (int) ((arr[i] / exp) % 10);
            output[--count[digit]] = arr[i];
        }
        return output;
    }
}
```

### Example

```
Sorted: [2, 24, 45, 66, 75, 90, 170, 802]
```

## 10. Bucket Sort

### Theory

Divide the value range into `k` buckets, drop each element into its bucket, sort each bucket individually, then concatenate. Best when values are roughly **uniformly distributed** — full theory and code already covered in [Lesson 9's original Bucket Idea section](#12-stability-in-place-comparators--partial-sorting) is now folded in here as one of the core ten; see the worked example below.

### Code

```java
import java.util.ArrayList;
import java.util.Arrays;
import java.util.Collections;
import java.util.List;

public class BucketSort {
    public static void main(String[] args) {
        double[] arr = { 0.42, 0.32, 0.23, 0.52, 0.25, 0.75 };
        System.out.println("Sorted: " + Arrays.toString(bucketSort(arr, 5)));
    }

    static double[] bucketSort(double[] arr, int bucketCount) {
        double min = Double.MAX_VALUE, max = -Double.MAX_VALUE;
        for (double value : arr) {
            min = Math.min(min, value);
            max = Math.max(max, value);
        }

        List<List<Double>> buckets = new ArrayList<>();
        for (int i = 0; i < bucketCount; i++) buckets.add(new ArrayList<>());

        double rangeSize = (max - min) / bucketCount;
        for (double value : arr) {
            int bucketIndex = (int) ((value - min) / rangeSize);
            if (bucketIndex == bucketCount) bucketIndex--;
            buckets.get(bucketIndex).add(value);
        }

        double[] result = new double[arr.length];
        int index = 0;
        for (List<Double> bucket : buckets) {
            Collections.sort(bucket);
            for (double value : bucket) result[index++] = value;
        }
        return result;
    }
}
```

### Example

```
Sorted: [0.23, 0.25, 0.32, 0.42, 0.52, 0.75]
```

## 11. Shell Sort

### Theory

An extension of insertion sort: instead of comparing only adjacent elements, compare elements a **gap** apart, shrinking the gap over successive passes until it reaches `1` (an ordinary insertion sort pass, but on an already mostly-sorted array by that point). Moving distant out-of-place elements early avoids the "many small shifts" cost that makes plain insertion sort slow on random data.

### Code

```java
import java.util.Arrays;

public class ShellSort {
    public static void main(String[] args) {
        int[] arr = { 5, 2, 8, 1, 9, 3, 7, 4 };
        shellSort(arr);
        System.out.println("Sorted: " + Arrays.toString(arr));
    }

    static void shellSort(int[] arr) {
        int n = arr.length;
        for (int gap = n / 2; gap > 0; gap /= 2) {
            for (int i = gap; i < n; i++) {
                int temp = arr[i];
                int j = i;
                while (j >= gap && arr[j - gap] > temp) {
                    arr[j] = arr[j - gap];
                    j -= gap;
                }
                arr[j] = temp;
            }
        }
    }
}
```

### Example

```
Sorted: [1, 2, 3, 4, 5, 7, 8, 9]
```

## 12. Stability, In-place, Comparators & Partial Sorting

### Theory

Four cross-cutting properties distinguish the ten algorithms above, and matter when picking one for a real problem:

- **Stable Sorting** — elements comparing as *equal* keep their original relative order. From the table: Bubble, Insertion, Merge, Counting, Radix, and Bucket sort are stable; Selection, Quick, Heap, and Shell sort are not. This matters whenever you sort by one key but want a secondary, implicit order preserved (e.g. records with equal scores keeping their original insertion order). Java's `Arrays.sort()` on **object arrays** uses stable Timsort; on **primitive arrays**, stability is meaningless anyway since identical primitives are indistinguishable.

  **Seeing instability concretely** — Selection Sort's swap-based approach can jump a later equal-key element in front of an earlier one:

  ```java
  public class InstabilityDemo {
      record Entry(int key, String label) {}

      public static void main(String[] args) {
          Entry[] arr = { new Entry(4, "a"), new Entry(2, "b"), new Entry(4, "c"), new Entry(2, "d") };
          selectionSortByKey(arr);
          System.out.println(java.util.Arrays.toString(arr));
      }

      static void selectionSortByKey(Entry[] arr) {
          for (int i = 0; i < arr.length; i++) {
              int minIndex = i;
              for (int j = i + 1; j < arr.length; j++) {
                  if (arr[j].key() < arr[minIndex].key()) minIndex = j;
              }
              Entry temp = arr[i];
              arr[i] = arr[minIndex];
              arr[minIndex] = temp;
          }
      }
  }
  ```

  ```
  [Entry[key=2, label=b], Entry[key=2, label=d], Entry[key=4, label=c], Entry[key=4, label=a]]
  ```

  Both `key=4` entries (`a` and `c`) started with `a` before `c` — after selection sort, `c` ends up before `a`. That flip is exactly what "unstable" means in practice.

- **In-place Sorting** — uses only O(1) extra memory (or O(log n) recursion stack). Bubble, Selection, Insertion, Quick, Heap, and Shell sort are in-place; Merge, Counting, Radix, and Bucket sort all need auxiliary space proportional to the input (or value range).

- **Custom Comparator** — sorting objects by a rule other than natural order requires a `Comparator`, only usable with object arrays (not raw primitive arrays):

```java
import java.util.Arrays;
import java.util.Comparator;

public class CustomComparator {
    public static void main(String[] args) {
        String[] words = { "banana", "kiwi", "apple", "fig" };
        Arrays.sort(words, Comparator.comparingInt(String::length));
        System.out.println("Sorted by length: " + Arrays.toString(words));
    }
}
```

  ```
  Sorted by length: [kiwi, fig, banana, apple]
  ```

- **Partial Sorting** — when only the top/bottom `k` elements are needed (not the full order), a heap of size `k` (see [Lesson 16: Heap + Arrays](<./16. Heap + Arrays.md#1-top-k>)) finds them in **O(n log k)** — faster than fully sorting with any algorithm above when `k` is small relative to `n`.

## 13. Tim Sort (Java's Real Default)

### Theory

None of the ten algorithms above is actually what `Arrays.sort()` runs when sorting **objects** (or what `Collections.sort()` runs on a `List`) — that's **Timsort**, a hybrid designed specifically to exploit "runs" of already-sorted data that show up constantly in real-world data:

1. Scan the array and identify existing **runs** (maximal already-ascending or already-descending stretches), reversing descending runs in place to make them ascending.
2. Extend any run shorter than a minimum size (~32-64 elements) using **insertion sort** — cheap for small ranges, and insertion sort is stable.
3. **Merge** runs together using the same merge step as [Merge Sort](#5-merge-sort), with a carefully tuned merge order that keeps the algorithm's memory usage and comparison count low.

The result: **O(n)** on already-sorted (or mostly-sorted) input, **O(n log n)** worst case, and — crucially for object sorting — **stable**, since both its building blocks (insertion sort, merge) are stable. This is exactly why Java picked it for anything involving a `Comparator`: correctness (stability) matters more for objects than the raw constant-factor speed of an unstable in-place sort.

> Dual-pivot Quicksort remains Java's choice for **primitive** arrays (`int[]`, `double[]`, ...) specifically *because* stability is meaningless for primitives, so Quicksort's better average-case performance wins with nothing to lose.

## 14. Multi-Key Sorting & Java's Collections.sort

### Theory

Real sorting needs often involve **multiple keys** — sort by one field, and break ties using another. Java's `Comparator` interface composes cleanly for this via `thenComparing`, and the same `Comparator` works whether you're calling `Arrays.sort()` on an object array or `Collections.sort()` / `list.sort()` on a `List`.

### Code

```java
import java.util.*;

public class MultiKeySorting {
    public static void main(String[] args) {
        String[] words = { "bb", "a", "cc", "d", "aa" };

        // Arrays.sort with a chained comparator: by length, then alphabetically
        Arrays.sort(words, Comparator.comparingInt(String::length).thenComparing(Comparator.naturalOrder()));
        System.out.println("Arrays.sort, length then alpha: " + Arrays.toString(words));

        // Collections.sort on a List, using the same kind of lambda/method-reference comparator
        List<String> list = new ArrayList<>(List.of("bb", "a", "cc", "d", "aa"));
        Collections.sort(list, (a, b) -> {
            if (a.length() != b.length()) return a.length() - b.length();
            return a.compareTo(b);
        });
        System.out.println("Collections.sort, equivalent lambda: " + list);

        // Modern idiom: List.sort() directly, no need for the Collections wrapper
        list.sort(Comparator.comparingInt(String::length).thenComparing(s -> s));
        System.out.println("list.sort (instance method): " + list);
    }
}
```

### Example

```
Arrays.sort, length then alpha: [a, d, aa, bb, cc]
Collections.sort, equivalent lambda: [a, d, aa, bb, cc]
list.sort (instance method): [a, d, aa, bb, cc]
```

All three produce the same result — `Arrays.sort(T[], Comparator)`, `Collections.sort(List<T>, Comparator)`, and `list.sort(Comparator)` are three entry points to the identical Timsort-based, stable, comparator-driven sort.

## 15. Which Sort Should You Actually Use?

### Theory

A short decision guide, tying the whole lesson together:

- **Sorting objects/a `List` in real Java code?** Just call `Arrays.sort()` / `Collections.sort()` / `list.sort()` — you get Timsort for free, stable and O(n log n) worst case. You almost never hand-roll a sort in production Java.
- **Sorting a primitive array (`int[]`, etc.) in real Java code?** `Arrays.sort()` again — Dual-Pivot Quicksort, unstable but fast, and stability doesn't apply to primitives anyway.
- **Small array (roughly < 20-50 elements) or nearly-sorted data?** Insertion Sort — simple, stable, and genuinely fast in this regime (this is also *why* Timsort uses insertion sort internally for small runs).
- **Need a guaranteed worst case, no O(n²) blowup risk?** Merge Sort (if you can afford O(n) extra space and want stability) or Heap Sort (if you need O(1) space and don't need stability).
- **Values are small integers with a known bound?** Counting Sort or Radix Sort — beats every comparison sort by not comparing at all.
- **Values are floats spread roughly evenly across a range?** Bucket Sort.
- **Minimizing memory writes specifically matters** (e.g. flash storage with limited write cycles)? Selection Sort's O(n) swaps are relevant here, though this is a niche concern outside typical DSA interview contexts.

## Quick Reference Cheat Sheet

| Algorithm | Best case | Worst case | Stable | In-place | Best used for |
|---|---|---|---|---|---|
| Bubble Sort | O(n) | O(n²) | Yes | Yes | teaching, nearly-sorted data |
| Selection Sort | O(n²) | O(n²) | No | Yes | minimizing swap count |
| Insertion Sort | O(n) | O(n²) | Yes | Yes | small or nearly-sorted arrays |
| Merge Sort | O(n log n) | O(n log n) | Yes | No | guaranteed O(n log n), stability needed |
| Quick Sort | O(n log n) | O(n²) | No | Yes | general purpose, average-case speed |
| Heap Sort | O(n log n) | O(n log n) | No | Yes | guaranteed O(n log n), O(1) space |
| Counting Sort | O(n+k) | O(n+k) | Yes | No | small integer range |
| Radix Sort | O(d(n+k)) | O(d(n+k)) | Yes | No | fixed-width integers, large n |
| Bucket Sort | O(n+k) | O(n²) | Yes | No | uniformly distributed floats |
| Shell Sort | O(n log n) | O(n²) | No | Yes | better than insertion sort, no extra space |
