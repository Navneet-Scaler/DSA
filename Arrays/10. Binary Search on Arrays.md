# 10. Binary Search on Arrays

Binary search cuts the search space in half at every step, turning an O(n) scan into **O(log n)** — but only on sorted data (or on the "monotonic answer space" pattern from [Maths Lesson 21](<../Maths/21. Binary Search on Answer.md>)). This lesson covers exact search plus the boundary-finding variants (lower/upper bound) that solve a much wider range of problems than exact match alone.

## 1. Basic Binary Search

### Theory

Given a sorted array, compare the target against the middle element: if equal, done; if the target is smaller, discard the right half; if larger, discard the left half. Repeat on the remaining half.

### Code

```java
public class BasicBinarySearch {
    public static void main(String[] args) {
        int[] arr = { 1, 3, 5, 7, 9, 11 };
        System.out.println("Index of 7: " + search(arr, 7));
    }

    static int search(int[] arr, int target) {
        int lo = 0, hi = arr.length - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (arr[mid] == target) return mid;
            else if (arr[mid] < target) lo = mid + 1;
            else hi = mid - 1;
        }
        return -1;
    }
}
```

### Example

```
Index of 7: 3
```

### Java's Built-in Binary Search

Hand-rolling binary search is for understanding the mechanics — in real code, reach for Java's built-ins:

- **`Arrays.binarySearch(int[] arr, int key)`** — for arrays, full coverage in [Lesson 3](<./03. Java Array APIs.md#5-arraysbinarysearch>).
- **`Collections.binarySearch(List<T> list, T key)`** — the `List` equivalent, same negative-result-encodes-insertion-point convention:

```java
import java.util.Collections;
import java.util.List;

public class CollectionsBinarySearchExample {
    public static void main(String[] args) {
        List<Integer> list = List.of(1, 3, 5, 7, 9, 11);
        System.out.println("Index of 7: " + Collections.binarySearch(list, 7));
    }
}
```

```
Index of 7: 3
```

Both require the input already **sorted** in the same order the search assumes — passing unsorted data silently produces a meaningless result rather than an error.

## 2. Lower Bound

### Theory

The **lower bound** of a target is the index of the **first** element that is `≥ target` — even if the exact value isn't present, this tells you where it would fit. Unlike exact search, this variant needs a **half-open** search space (`hi = length`, not `length - 1`) so it can express "the answer is past every element."

### Code

```java
public class LowerBound {
    public static void main(String[] args) {
        int[] arr = { 1, 2, 2, 2, 3, 4 };
        System.out.println("Lower bound of 2: " + lowerBound(arr, 2));
    }

    static int lowerBound(int[] arr, int target) {
        int lo = 0, hi = arr.length;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (arr[mid] < target) lo = mid + 1;
            else hi = mid;
        }
        return lo;
    }
}
```

### Example

```
Lower bound of 2: 1
```

Index `1` is the **first** occurrence of `2` in `[1, 2, 2, 2, 3, 4]`.

## 3. Upper Bound

### Theory

The **upper bound** is the index of the **first** element that is **strictly greater** than the target — one past the last occurrence of the target itself. Same template as [Lower Bound](#2-lower-bound), just with `<=` instead of `<` in the comparison.

### Code

```java
public class UpperBound {
    public static void main(String[] args) {
        int[] arr = { 1, 2, 2, 2, 3, 4 };
        System.out.println("Upper bound of 2: " + upperBound(arr, 2));
    }

    static int upperBound(int[] arr, int target) {
        int lo = 0, hi = arr.length;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (arr[mid] <= target) lo = mid + 1;
            else hi = mid;
        }
        return lo;
    }
}
```

### Example

```
Upper bound of 2: 4
```

Index `4` is where `3` sits — one past the last `2`. `upperBound - lowerBound` gives the **count** of the target in the array (here, `4 - 1 = 3` occurrences of `2`).

## 4. Search Insert Position

### Theory

"Where would this target be inserted to keep the array sorted?" is **exactly** the [Lower Bound](#2-lower-bound) query — if the target exists, its insert position is its first occurrence; if it doesn't exist, it's the index of the first larger element (functionally identical code).

### Code

```java
public class SearchInsertPosition {
    public static void main(String[] args) {
        int[] arr = { 1, 3, 5, 6 };
        System.out.println("Insert position for 5: " + searchInsert(arr, 5));
        System.out.println("Insert position for 2: " + searchInsert(arr, 2));
        System.out.println("Insert position for 7: " + searchInsert(arr, 7));
    }

    static int searchInsert(int[] arr, int target) {
        int lo = 0, hi = arr.length;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (arr[mid] < target) lo = mid + 1;
            else hi = mid;
        }
        return lo;
    }
}
```

### Example

```
Insert position for 5: 2
Insert position for 2: 1
Insert position for 7: 4
```

## 5. Binary Search on Answer

### Theory

Full theory and derivation is in [Maths Lesson 21: Binary Search on Answer](<../Maths/21. Binary Search on Answer.md>) — the same "monotonic condition" idea applies directly to array problems where the search space isn't the array's *indices*, but some other value entirely (a capacity, a speed, a maximum). The array itself is just an input to the monotonic **check function**, not the thing being binary-searched.

**Array-specific example — minimum array-sum-splitting capacity:** given an array and a number of groups `k`, find the smallest "maximum subarray sum" achievable by splitting the array into `k` contiguous parts. As the candidate capacity increases, the number of parts needed to stay under it **decreases monotonically** — a textbook binary-search-on-answer setup.

### Problem (Medium): Koko Eating Bananas

**Theory:** Full theory in [Maths Lesson 21](<../Maths/21. Binary Search on Answer.md#3-search-over-a-mathematical-function>) — repeated here with code since it's one of the most commonly asked array-flavored binary-search-on-answer problems. Koko must finish all banana piles within `h` hours, eating at a constant speed `k` bananas/hour; find the minimum speed. As speed increases, total eating time only ever decreases — the monotonic condition binary search needs.

```java
public class KokoEatingBananas {
    public static void main(String[] args) {
        int[] piles = { 3, 6, 7, 11 };
        System.out.println("Minimum eating speed: " + minEatingSpeed(piles, 8));
    }

    static int minEatingSpeed(int[] piles, int h) {
        int lo = 1, hi = 0;
        for (int pile : piles) hi = Math.max(hi, pile);

        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (hoursNeeded(piles, mid) <= h) {
                hi = mid;
            } else {
                lo = mid + 1;
            }
        }
        return lo;
    }

    static long hoursNeeded(int[] piles, int speed) {
        long hours = 0;
        for (int pile : piles) {
            hours += (pile + speed - 1) / speed; // ceiling division
        }
        return hours;
    }
}
```

```
Minimum eating speed: 4
```

## 6. Rotated Array Search

### Theory

A **rotated sorted array** (e.g. `[4,5,6,7,0,1,2]`, originally `[0,1,2,4,5,6,7]` rotated) is no longer globally sorted, but at every step, **at least one half around the midpoint is still sorted**. Binary search still works: check which half is sorted, then decide whether the target could be in that sorted half's range — if not, it must be in the other half.

### Code

```java
public class RotatedArraySearch {
    public static void main(String[] args) {
        int[] arr = { 4, 5, 6, 7, 0, 1, 2 };
        System.out.println("Index of 0: " + search(arr, 0));
    }

    static int search(int[] arr, int target) {
        int lo = 0, hi = arr.length - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (arr[mid] == target) return mid;

            if (arr[lo] <= arr[mid]) { // left half is sorted
                if (arr[lo] <= target && target < arr[mid]) {
                    hi = mid - 1;
                } else {
                    lo = mid + 1;
                }
            } else { // right half is sorted
                if (arr[mid] < target && target <= arr[hi]) {
                    lo = mid + 1;
                } else {
                    hi = mid - 1;
                }
            }
        }
        return -1;
    }
}
```

### Example

```
Index of 0: 4
```

### Problem 2 (Advanced): Search in Rotated Sorted Array II (with duplicates)

**Theory:** The same rotated search, but duplicates are now allowed — which breaks the "at least one half is sorted" guarantee: if `arr[lo] == arr[mid] == arr[hi]`, there's no way to tell which side is sorted just from those three values (e.g. `[1,1,1,0,1,1,1]`). The fix: when this ambiguous case hits, just shrink both ends by one and try again — this can degrade the algorithm to O(n) in the worst case (all duplicates), but keeps it correct.

```java
public class SearchRotatedWithDuplicates {
    public static void main(String[] args) {
        System.out.println("Found 0 in [2,5,6,0,0,1,2]: " + search(new int[] { 2, 5, 6, 0, 0, 1, 2 }, 0));
        System.out.println("Found 0 in [1,0,1,1,1]: " + search(new int[] { 1, 0, 1, 1, 1 }, 0));
    }

    static boolean search(int[] nums, int target) {
        int lo = 0, hi = nums.length - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (nums[mid] == target) return true;

            if (nums[lo] == nums[mid] && nums[mid] == nums[hi]) {
                lo++; // ambiguous -- can't tell which half is sorted, shrink both ends
                hi--;
            } else if (nums[lo] <= nums[mid]) {
                if (nums[lo] <= target && target < nums[mid]) hi = mid - 1;
                else lo = mid + 1;
            } else {
                if (nums[mid] < target && target <= nums[hi]) lo = mid + 1;
                else hi = mid - 1;
            }
        }
        return false;
    }
}
```

```
Found 0 in [2,5,6,0,0,1,2]: true
Found 0 in [1,0,1,1,1]: true
```

## 7. Find Peak Element

### Theory

A "peak" is any element strictly greater than its neighbors (array boundaries count as `-∞`). The array need **not** be sorted — but comparing `arr[mid]` to `arr[mid+1]` still gives a valid binary-search direction: if `arr[mid] < arr[mid+1]`, a peak must exist somewhere to the right (the sequence is still climbing); otherwise, a peak exists at `mid` or somewhere to its left. This works because **some** peak is guaranteed to exist in whichever direction values are currently increasing.

### Code

```java
public class FindPeakElement {
    public static void main(String[] args) {
        System.out.println("Peak index [1,2,3,1]: " + findPeakElement(new int[] { 1, 2, 3, 1 }));
        System.out.println("Peak index [1,2,1,3,5,6,4]: " + findPeakElement(new int[] { 1, 2, 1, 3, 5, 6, 4 }));
    }

    static int findPeakElement(int[] nums) {
        int lo = 0, hi = nums.length - 1;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (nums[mid] > nums[mid + 1]) {
                hi = mid; // currently descending -- a peak is at mid or to its left
            } else {
                lo = mid + 1; // currently ascending -- a peak must exist to the right
            }
        }
        return lo;
    }
}
```

### Example

```
Peak index [1,2,3,1]: 2
Peak index [1,2,1,3,5,6,4]: 5
```

## 8. Median of Two Sorted Arrays

### Theory

The most advanced binary search in this lesson: find the median of two sorted arrays combined, in **O(log(min(m, n)))** — without merging them. The trick is binary searching for a **partition point** in the smaller array such that the corresponding partition in the other array balances the total left/right element counts, and every element on the "left" side of both partitions is `≤` every element on the "right" side. Once such a balanced partition is found, the median comes directly from the four boundary values around it — no merge required.

### Code

```java
public class MedianOfTwoSortedArrays {
    public static void main(String[] args) {
        System.out.println("Median [1,3] & [2]: " + findMedianSortedArrays(new int[] { 1, 3 }, new int[] { 2 }));
        System.out.println("Median [1,2] & [3,4]: " + findMedianSortedArrays(new int[] { 1, 2 }, new int[] { 3, 4 }));
    }

    static double findMedianSortedArrays(int[] a, int[] b) {
        if (a.length > b.length) return findMedianSortedArrays(b, a); // ensure 'a' is the smaller array

        int m = a.length, n = b.length;
        int lo = 0, hi = m;
        int half = (m + n + 1) / 2;

        while (lo <= hi) {
            int i = lo + (hi - lo) / 2; // partition index in a
            int j = half - i;           // matching partition index in b

            int leftA = (i == 0) ? Integer.MIN_VALUE : a[i - 1];
            int rightA = (i == m) ? Integer.MAX_VALUE : a[i];
            int leftB = (j == 0) ? Integer.MIN_VALUE : b[j - 1];
            int rightB = (j == n) ? Integer.MAX_VALUE : b[j];

            if (leftA <= rightB && leftB <= rightA) {
                if ((m + n) % 2 == 1) {
                    return Math.max(leftA, leftB);
                }
                return (Math.max(leftA, leftB) + Math.min(rightA, rightB)) / 2.0;
            } else if (leftA > rightB) {
                hi = i - 1; // partition in a is too far right
            } else {
                lo = i + 1; // partition in a is too far left
            }
        }
        throw new IllegalArgumentException("Input arrays are not sorted");
    }
}
```

### Example

```
Median [1,3] & [2]: 2.0
Median [1,2] & [3,4]: 2.5
```

## Quick Reference Cheat Sheet

| Variant | Finds | Search space |
|---|---|---|
| Basic Binary Search | exact index of target | `[0, length-1]` |
| Lower Bound | first index `≥ target` | `[0, length]` (half-open) |
| Upper Bound | first index `> target` | `[0, length]` (half-open) |
| Search Insert Position | same as Lower Bound | `[0, length]` |
| Binary Search on Answer | smallest/largest value satisfying a monotonic check (e.g. Koko Eating Bananas) | whatever range the answer could be in |
| Rotated Array Search | target in a rotated sorted array, with or without duplicates | `[0, length-1]`, checking which half is sorted |
| Find Peak Element | any local maximum, even in an unsorted array | `[0, length-1]`, direction from comparing neighbors |
| Median of Two Sorted Arrays | balanced partition across both arrays | `[0, length of smaller array]` |
