# 16. Heap + Arrays

A heap keeps its minimum (or maximum) element accessible in **O(1)**, with **O(log k)** insertion and removal — the right tool whenever a problem only cares about the top/bottom `k` elements of a stream or array, rather than the full sorted order. This lesson covers Java's `PriorityQueue` applied to the classic top-k, running-median, and windowed-maximum problems.

## 1. Top K

### Theory

The general pattern: maintain a heap of size **exactly `k`**, and for every new element, add it and — if the heap now exceeds size `k` — evict whichever element is least relevant to keep. This finds the top `k` elements in **O(n log k)**, beating a full O(n log n) sort whenever `k` is much smaller than `n` (mentioned as [Partial Sorting](<./09. Sorting Based Arrays.md#5-partial-sorting>) in Lesson 9).

## 2. K Largest

### Theory

To find the `k` **largest** elements, use a **min-heap** of size `k`: push every element, and whenever the heap exceeds size `k`, pop the minimum. Counterintuitively, a *min*-heap finds the *largest* elements — because the smallest of the "top k so far" is exactly the one that should be evicted the moment a bigger candidate shows up.

### Code

```java
import java.util.PriorityQueue;
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class KLargest {
    public static void main(String[] args) {
        int[] arr = { 3, 2, 1, 5, 6, 4 };
        System.out.println("2 largest: " + kLargest(arr, 2));
    }

    static List<Integer> kLargest(int[] arr, int k) {
        PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // natural order = min-heap
        for (int value : arr) {
            minHeap.offer(value);
            if (minHeap.size() > k) {
                minHeap.poll(); // evict the current smallest of the "top k so far"
            }
        }
        List<Integer> result = new ArrayList<>(minHeap);
        Collections.sort(result);
        return result;
    }
}
```

### Example

```
2 largest: [5, 6]
```

## 3. K Smallest

### Theory

The mirror of [K Largest](#2-k-largest): use a **max-heap** of size `k` instead (in Java, `PriorityQueue<>(Collections.reverseOrder())`), evicting the current **maximum** whenever the heap exceeds size `k`.

### Code

```java
import java.util.PriorityQueue;
import java.util.Collections;
import java.util.ArrayList;
import java.util.List;

public class KSmallest {
    public static void main(String[] args) {
        int[] arr = { 3, 2, 1, 5, 6, 4 };
        System.out.println("2 smallest: " + kSmallest(arr, 2));
    }

    static List<Integer> kSmallest(int[] arr, int k) {
        PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
        for (int value : arr) {
            maxHeap.offer(value);
            if (maxHeap.size() > k) {
                maxHeap.poll(); // evict the current largest of the "bottom k so far"
            }
        }
        List<Integer> result = new ArrayList<>(maxHeap);
        Collections.sort(result);
        return result;
    }
}
```

### Example

```
2 smallest: [1, 2]
```

## 4. Running Median

### Theory

To track the **median of a growing stream** efficiently, split the data across **two heaps**: a max-heap holding the smaller half, and a min-heap holding the larger half, kept balanced in size (differing by at most 1). The median is then either the max-heap's top (odd total count) or the average of both heaps' tops (even count) — no re-sorting needed as new values arrive.

### Code

```java
import java.util.PriorityQueue;
import java.util.Collections;

public class RunningMedian {
    private PriorityQueue<Integer> lowerHalf = new PriorityQueue<>(Collections.reverseOrder()); // max-heap
    private PriorityQueue<Integer> upperHalf = new PriorityQueue<>(); // min-heap

    public static void main(String[] args) {
        RunningMedian rm = new RunningMedian();
        for (int num : new int[] { 5, 15, 1, 3 }) {
            rm.add(num);
            System.out.println("After adding " + num + ", median = " + rm.median());
        }
    }

    void add(int num) {
        lowerHalf.offer(num);
        upperHalf.offer(lowerHalf.poll()); // always route through lowerHalf first to keep balance rules simple
        if (upperHalf.size() > lowerHalf.size()) {
            lowerHalf.offer(upperHalf.poll());
        }
    }

    double median() {
        if (lowerHalf.size() > upperHalf.size()) {
            return lowerHalf.peek();
        }
        return (lowerHalf.peek() + upperHalf.peek()) / 2.0;
    }
}
```

### Example

```
After adding 5, median = 5.0
After adding 15, median = 10.0
After adding 1, median = 5.0
After adding 3, median = 4.0
```

## 5. Sliding Window Maximum

### Theory

The primary O(n) solution — a **monotonic deque** — is covered in full in [Lesson 7: Maximum Window](<./07. Sliding Window.md#5-maximum-window>). A **heap-based** alternative also works, at the cost of extra complexity: push `(value, index)` pairs into a max-heap, and whenever the heap's top index has fallen outside the current window, **lazily discard** it (rather than proactively removing it) before reading the max. This "lazy deletion" trick avoids needing a heap that supports arbitrary removal, but runs in **O(n log n)** — worse than the deque's O(n), so the deque should be your default. This entry exists mainly to show *why* Heap + Arrays problems don't automatically make every windowed problem a heap problem.

### Code

```java
import java.util.PriorityQueue;
import java.util.Collections;
import java.util.List;
import java.util.ArrayList;

public class SlidingWindowMaximumHeap {
    public static void main(String[] args) {
        int[] arr = { 1, 3, -1, -3, 5, 3, 6, 7 };
        System.out.println("Sliding window maximums (heap, k=3): " + slidingWindowMax(arr, 3));
    }

    static List<Integer> slidingWindowMax(int[] arr, int k) {
        // Max-heap of [value, index] pairs, ordered by value descending
        PriorityQueue<int[]> maxHeap = new PriorityQueue<>((a, b) -> b[0] - a[0]);
        List<Integer> result = new ArrayList<>();

        for (int i = 0; i < arr.length; i++) {
            maxHeap.offer(new int[] { arr[i], i });

            while (maxHeap.peek()[1] <= i - k) {
                maxHeap.poll(); // lazily discard entries that fell out of the window
            }
            if (i >= k - 1) {
                result.add(maxHeap.peek()[0]);
            }
        }
        return result;
    }
}
```

### Example

```
Sliding window maximums (heap, k=3): [3, 3, 5, 5, 6, 7]
```

Same result as the deque-based version in [Lesson 7](<./07. Sliding Window.md#5-maximum-window>) — just arrived at with a different (and slower) tool.

## Quick Reference Cheat Sheet

| Problem | Heap type | Complexity |
|---|---|---|
| K Largest | min-heap of size `k` | O(n log k) |
| K Smallest | max-heap of size `k` | O(n log k) |
| Running Median | max-heap (lower half) + min-heap (upper half) | O(log n) per insert |
| Sliding Window Maximum (heap version) | max-heap with lazy deletion | O(n log n) — prefer the deque approach |
