# 11. Decimal Mathematics

Working with decimal numbers looks simple until precision limits, rounding rules, and floating-point representation quirks start producing results that don't match hand calculation. This lesson explains *why* those quirks happen and gives the Java tools (`BigDecimal`, `RoundingMode`, `MathContext`) to control them precisely.

## 1. Precision

### Theory

Java has two floating-point primitive types, both IEEE-754 binary floating point:

| Type | Size | Approximate decimal precision |
|---|---|---|
| `float` | 32-bit | ~6–7 significant decimal digits |
| `double` | 64-bit | ~15–17 significant decimal digits |

Both trade exactness for range and speed — they store numbers as a sign, an exponent, and a fixed number of binary mantissa bits, so **not every decimal value has an exact binary representation** (see the next section). When exact decimal precision matters — money, measurements, anything audited — use `java.math.BigDecimal` instead, which stores digits exactly and lets *you* control rounding.

### Code

```java
import java.math.BigDecimal;

public class Precision {
    public static void main(String[] args) {
        float f = 1.0f / 3;
        double d = 1.0 / 3;
        BigDecimal bd = BigDecimal.ONE.divide(BigDecimal.valueOf(3), 20, java.math.RoundingMode.HALF_UP);

        System.out.println("1/3 as float:      " + f);
        System.out.println("1/3 as double:     " + d);
        System.out.println("1/3 as BigDecimal (20 digits): " + bd);
    }
}
```

### Example

```
1/3 as float:      0.33333334
1/3 as double:     0.3333333333333333
1/3 as BigDecimal (20 digits): 0.33333333333333333333
```

## 2. Floating Point Errors

### Theory

Binary floating point can only represent numbers that are sums of powers of 2 exactly. Just like `1/3` has no exact finite decimal representation, many simple decimals (`0.1`, `0.2`, `0.3`, ...) have **no exact finite binary representation** — they're stored as the *closest possible* `double`, which introduces a tiny rounding error. That error can surface when you do arithmetic:

```
0.1 + 0.2 == 0.30000000000000004   (not exactly 0.3)
```

> **Golden rule:** never compare `double`s with `==`. Instead, check whether the difference is smaller than a small tolerance (`epsilon`), or use `BigDecimal` if you need exact decimal comparisons.

### Code

```java
public class FloatingPointErrors {
    public static void main(String[] args) {
        double sum = 0.1 + 0.2;
        System.out.println("0.1 + 0.2 = " + sum);
        System.out.println("0.1 + 0.2 == 0.3 ? " + (sum == 0.3)); // false!

        // Safe comparison using an epsilon tolerance
        double epsilon = 1e-9;
        boolean approximatelyEqual = Math.abs(sum - 0.3) < epsilon;
        System.out.println("Approximately equal (epsilon check): " + approximatelyEqual);
    }
}
```

### Example

```
0.1 + 0.2 = 0.30000000000000004
0.1 + 0.2 == 0.3 ? false
Approximately equal (epsilon check): true
```

## 3. Rounding

### Theory

Java offers several rounding tools, and they don't all agree on how to handle exact halfway values (`x.5`):

- **`Math.round(double)`** computes `floor(x + 0.5)` — it rounds halfway values **toward positive infinity**, not "away from zero". This means `Math.round(2.5) == 3`, but `Math.round(-2.5) == -2` (not `-3`) — a common surprise.
- **`BigDecimal` with a `RoundingMode`** gives explicit control:
  - `RoundingMode.HALF_UP` — rounds halfway values away from zero (the "textbook" rounding most people expect).
  - `RoundingMode.HALF_EVEN` ("banker's rounding") — rounds halfway values to the nearest **even** digit, reducing cumulative bias when rounding many values. This is the default in many financial systems.

### Code

```java
import java.math.BigDecimal;
import java.math.RoundingMode;

public class Rounding {
    public static void main(String[] args) {
        // Math.round's negative-halfway gotcha
        System.out.println("Math.round(2.5):  " + Math.round(2.5));
        System.out.println("Math.round(-2.5): " + Math.round(-2.5)); // -2, not -3!

        // HALF_UP vs HALF_EVEN
        System.out.println("2.5 HALF_UP:   " + round(2.5, 0, RoundingMode.HALF_UP));
        System.out.println("2.5 HALF_EVEN: " + round(2.5, 0, RoundingMode.HALF_EVEN));
        System.out.println("3.5 HALF_UP:   " + round(3.5, 0, RoundingMode.HALF_UP));
        System.out.println("3.5 HALF_EVEN: " + round(3.5, 0, RoundingMode.HALF_EVEN));

        // Rounding to N decimal places
        System.out.println("3.14159 to 2dp (HALF_UP): " + round(3.14159, 2, RoundingMode.HALF_UP));
    }

    static BigDecimal round(double value, int scale, RoundingMode mode) {
        return BigDecimal.valueOf(value).setScale(scale, mode);
    }
}
```

### Example

```
Math.round(2.5):  3
Math.round(-2.5): -2
2.5 HALF_UP:   3
2.5 HALF_EVEN: 2
3.5 HALF_UP:   4
3.5 HALF_EVEN: 4
3.14159 to 2dp (HALF_UP): 3.14
```

## 4. Significant Digits

### Theory

Significant digits (or "significant figures") are the digits in a number that carry real precision. Rules for counting them:

- All **non-zero digits** are significant.
- Zeros **between** non-zero digits are significant (`105` has 3 sig figs).
- **Leading** zeros are never significant (`0.0045` has 2 sig figs: `4` and `5`).
- **Trailing** zeros after a decimal point are significant (`45.00` has 4 sig figs).
- Trailing zeros with **no decimal point** are ambiguous (`4500` could have 2, 3, or 4 sig figs) — scientific notation resolves this (see next section).

To round a value to a specific number of significant digits in Java, use `BigDecimal` with a `MathContext`.

### Code

```java
import java.math.BigDecimal;
import java.math.MathContext;
import java.math.RoundingMode;

public class SignificantDigits {
    public static void main(String[] args) {
        BigDecimal value = new BigDecimal("123456.789");

        BigDecimal threeSigFigs = value.round(new MathContext(3, RoundingMode.HALF_UP));
        BigDecimal fiveSigFigs = value.round(new MathContext(5, RoundingMode.HALF_UP));

        System.out.println("123456.789 to 3 sig figs: " + threeSigFigs);
        System.out.println("123456.789 to 5 sig figs: " + fiveSigFigs);
    }
}
```

### Example

```
123456.789 to 3 sig figs: 1.23E+5
123456.789 to 5 sig figs: 1.2346E+5
```

## 5. Scientific Notation

### Theory

Scientific notation writes a number as a single digit (1–9) times a power of 10: `123456.789 = 1.23456789 × 10^5`. It's the standard way to express very large or very small numbers unambiguously — and it resolves the trailing-zero significant-digit ambiguity mentioned above (`4.5 × 10^3` unambiguously has 2 sig figs, unlike plain `4500`).

Java's `String.format("%e", ...)` produces scientific notation directly (6 digits after the decimal point by default; use `%.Ne` to control precision), and `Double.parseDouble` reads scientific-notation strings back in.

### Code

```java
public class ScientificNotation {
    public static void main(String[] args) {
        double value = 123456.789;

        System.out.println("Default %e:  " + String.format("%e", value));
        System.out.println("2 digits %.2e: " + String.format("%.2e", value));

        // Parsing scientific notation back to a double
        double avogadro = Double.parseDouble("6.022e23");
        System.out.println("Parsed 6.022e23: " + avogadro);
    }
}
```

### Example

```
Default %e:  1.234568e+05
2 digits %.2e: 1.23e+05
Parsed 6.022e23: 6.022E23
```

## Quick Reference Cheat Sheet

| Concept | Key takeaway |
|---|---|
| Precision | `float` ~7 digits, `double` ~15–17 digits; use `BigDecimal` for exact decimals |
| Floating Point Errors | binary can't represent every decimal exactly — never compare `double`s with `==` |
| `Math.round` | rounds via `floor(x + 0.5)` — halfway values go toward `+∞`, so `-2.5 → -2` |
| `RoundingMode.HALF_UP` | rounds halfway values away from zero (textbook rounding) |
| `RoundingMode.HALF_EVEN` | rounds halfway values to the nearest even digit (banker's rounding) |
| Significant digits | leading zeros don't count, trailing zeros after a decimal point do |
| Scientific notation | `String.format("%e", x)`, parse back with `Double.parseDouble` |
