## 1. Greedy + Math

### Theory

A greedy algorithm makes the locally-best choice at each step, hoping it leads to a globally-best result. That "hope" needs a **mathematical justification** — usually an **exchange argument**: show that any optimal solution can be rearranged to match the greedy choice without getting worse. **Classic example — coin change with canonical denominations** (`{1, 5, 10, 25}`, like US coins): always take as many of the largest coin as possible. This provably gives the minimum coin count *for this specific denomination set*, because each coin is more than double the next-smaller one — but the same greedy strategy **fails** for arbitrary denominations (e.g. `{1, 3, 4}` making `6`: greedy picks `4+1+1` = 3 coins, but `3+3` = 2 coins is better). Always verify the mathematical property that makes greedy valid — don't assume it transfers to a different-looking problem.

### Code

```java
import java.util.ArrayList;
import java.util.List;

public class GreedyMath {
    public static void main(String[] args) {
        int[] coins = { 25, 10, 5, 1 }; // canonical denominations, largest first
        int amount = 63;

        List<int[]> used = greedyCoinChange(amount, coins);
        System.out.println("Greedy coin breakdown for " + amount + ":");
        int totalCoins = 0;
        for (int[] pair : used) {
            System.out.println("  " + pair[1] + " x " + pair[0]);
            totalCoins += pair[1];
        }
        System.out.println("Total coins used: " + totalCoins);
    }

    static List<int[]> greedyCoinChange(int amount, int[] coins) {
        List<int[]> used = new ArrayList<>();
        for (int coin : coins) {
            int count = amount / coin;
            if (count > 0) {
                used.add(new int[] { coin, count });
                amount -= coin * count;
            }
        }
        return used;
    }
}
```

### Example

```
Greedy coin breakdown for 63:
  2 x 25
  1 x 10
  3 x 1
Total coins used: 6
```

### Problem (Medium): Fractional Knapsack

**Theory:** **Classic example — Fractional Knapsack:** given items with values and weights, and a knapsack of limited capacity, maximize total value, where items **can be split** (unlike 0/1 Knapsack, which needs DP because whole-item-only decisions aren't exchangeable this simply). The exchange argument here is clean: sort items by **value-per-weight ratio**, descending, and greedily take as much of each as fits — if an optimal solution ever preferred a lower-ratio item's weight over some of a higher-ratio item, swapping that weight to the higher-ratio item strictly improves the total value, contradicting optimality. Being able to take *fractions* is exactly what makes this exchange always possible (0/1 Knapsack has no such freedom, which is why its greedy analogue fails).

```java
import java.util.Arrays;
import java.util.Comparator;

public class FractionalKnapsack {
    public static void main(String[] args) {
        int[] values = { 60, 100, 120 };
        int[] weights = { 10, 20, 30 };
        int capacity = 50;

        System.out.println("Max value (fractional knapsack): " + fractionalKnapsack(values, weights, capacity));
    }

    static double fractionalKnapsack(int[] values, int[] weights, int capacity) {
        int n = values.length;
        Integer[] indices = new Integer[n];
        for (int i = 0; i < n; i++) indices[i] = i;

        Arrays.sort(indices, Comparator.comparingDouble((Integer i) -> (double) values[i] / weights[i]).reversed());

        double totalValue = 0;
        int remaining = capacity;
        for (int i : indices) {
            if (remaining >= weights[i]) {
                totalValue += values[i];
                remaining -= weights[i];
            } else {
                totalValue += values[i] * ((double) remaining / weights[i]);
                break;
            }
        }
        return totalValue;
    }
}
```

```
Max value (fractional knapsack): 240.0
```

Ratios: item 1 (`60/10=6`), item 2 (`100/20=5`), item 3 (`120/30=4`). Greedily: take all of item 1 (value `60`, `40` capacity left), all of item 2 (value `100`, `20` capacity left), then `2/3` of item 3 (value `80`) — total `240`, the provably optimal split.

## 2. Observation

### Theory

Some problems become trivial once you notice the right **pattern**, without any formal derivation up front. **Classic example — XOR of all numbers from `1` to `n`:** computing it naively is `O(n)`, but the result cycles with period `4` based on `n mod 4`:

| `n mod 4` | XOR(1..n) |
|---|---|
| `0` | `n` |
| `1` | `1` |
| `2` | `n + 1` |
| `3` | `0` |

This observation turns an `O(n)` loop into an `O(1)` lookup — the kind of shortcut worth testing for (with small brute-force cases) whenever a problem asks for a cumulative XOR, sum, or similar operation over a range.

### Code

```java
public class Observation {
    public static void main(String[] args) {
        for (int n : new int[] { 1, 4, 5, 7, 8, 10 }) {
            System.out.println("XOR(1.." + n + ") brute=" + xorBrute(n) + ", formula=" + xorFormula(n));
        }
    }

    static int xorBrute(int n) {
        int result = 0;
        for (int i = 1; i <= n; i++) result ^= i;
        return result;
    }

    static int xorFormula(int n) {
        switch (n % 4) {
            case 0: return n;
            case 1: return 1;
            case 2: return n + 1;
            default: return 0; // n % 4 == 3
        }
    }
}
```

### Example

```
XOR(1..1) brute=1, formula=1
XOR(1..4) brute=4, formula=4
XOR(1..5) brute=1, formula=1
XOR(1..7) brute=0, formula=0
XOR(1..8) brute=8, formula=8
XOR(1..10) brute=11, formula=11
```

### Problem (Basic): Nim Game

**Theory:** **Classic example — Nim Game:** two players alternately remove `1`, `2`, or `3` stones from a pile of `n`; whoever takes the last stone wins. Brute-force game-tree search (memoized on "does the current player win with `n` stones left?") reveals a striking pattern by testing small cases: the current player **loses** exactly when `n` is a multiple of `4` — because whatever they take (`1`, `2`, or `3`), the opponent can always take enough to bring the pile back down to the next-lower multiple of `4`, eventually forcing the losing player to face `n=0`. This turns an exponential game-tree search into an `O(1)` check.

```java
import java.util.HashMap;
import java.util.Map;

public class NimGame {
    public static void main(String[] args) {
        for (int n : new int[] { 1, 2, 3, 4, 5, 8, 9 }) {
            System.out.println("canWinNim(" + n + ") brute=" + canWinNimBrute(n, new HashMap<>())
                    + " formula=" + canWinNimFormula(n));
        }
    }

    // Exponential without memoization -- included to show where the pattern comes from
    static boolean canWinNimBrute(int n, Map<Integer, Boolean> memo) {
        if (n <= 0) return false;
        if (memo.containsKey(n)) return memo.get(n);

        boolean result = false;
        for (int take = 1; take <= 3 && take <= n; take++) {
            if (!canWinNimBrute(n - take, memo)) {
                result = true;
                break;
            }
        }
        memo.put(n, result);
        return result;
    }

    // O(1), from the observed pattern
    static boolean canWinNimFormula(int n) {
        return n % 4 != 0;
    }
}
```

```
canWinNim(1) brute=true formula=true
canWinNim(2) brute=true formula=true
canWinNim(3) brute=true formula=true
canWinNim(4) brute=false formula=false
canWinNim(5) brute=true formula=true
canWinNim(8) brute=false formula=false
canWinNim(9) brute=true formula=true
```

At `n=4`: whatever the current player takes (`1`, `2`, or `3`), the pile becomes `3`, `2`, or `1` — all winning positions for the *opponent* — so `4` is a guaranteed loss, matching `4 % 4 == 0`.

## 3. Mathematical Proof

### Theory

Before trusting a formula (especially one found via [Observation](#2-observation)), prove it — **induction** is the standard tool for statements about all positive integers `n`:

1. **Base case:** verify the statement for the smallest `n` (usually `n=1`).
2. **Inductive step:** assume it holds for `n=k`, then prove it must hold for `n=k+1`.

**Classic example — the sum of the first `n` odd numbers is `n²`:**

```
Base case (n=1): sum = 1 = 1²  ✓
Inductive step: assume 1+3+...+(2k-1) = k².
  Then 1+3+...+(2k-1)+(2k+1) = k² + (2k+1) = (k+1)²  ✓
```

Since both parts hold, the formula is true for **all** `n` by induction.

### Code

```java
public class MathematicalProof {
    public static void main(String[] args) {
        for (int n : new int[] { 1, 2, 3, 4, 5, 10 }) {
            System.out.println("Sum of first " + n + " odd numbers: " + sumOfOdds(n)
                    + " (n^2 = " + (n * n) + ")");
        }
    }

    static int sumOfOdds(int n) {
        int sum = 0;
        for (int i = 1; i <= n; i++) {
            sum += (2 * i - 1); // the i-th odd number
        }
        return sum;
    }
}
```

### Example

```
Sum of first 1 odd numbers: 1 (n^2 = 1)
Sum of first 2 odd numbers: 4 (n^2 = 4)
Sum of first 3 odd numbers: 9 (n^2 = 9)
Sum of first 4 odd numbers: 16 (n^2 = 16)
Sum of first 5 odd numbers: 25 (n^2 = 25)
Sum of first 10 odd numbers: 100 (n^2 = 100)
```

The loop-based sum always matches `n²` exactly — consistent with the induction proof above.

## 4. Formula Derivation

### Theory

Deriving a closed-form formula from a pattern or recurrence avoids an `O(n)` loop entirely. **Classic example — sum of the first `n` squares.** One derivation approach uses the identity `(k+1)³ - k³ = 3k² + 3k + 1`, summed over `k = 1` to `n`; the left side telescopes to `(n+1)³ - 1`, and solving for `Σk²` gives:

```
Σ(k²) for k=1 to n  =  n(n+1)(2n+1) / 6
```

Once derived, this replaces an `O(n)` summation loop with an **O(1)** computation.

### Code

```java
public class FormulaDerivation {
    public static void main(String[] args) {
        for (int n : new int[] { 1, 5, 10, 20 }) {
            System.out.println("Sum of squares 1.." + n + " brute=" + sumSquaresBrute(n)
                    + ", formula=" + sumSquaresFormula(n));
        }
    }

    static long sumSquaresBrute(int n) {
        long sum = 0;
        for (int i = 1; i <= n; i++) sum += (long) i * i;
        return sum;
    }

    static long sumSquaresFormula(int n) {
        return (long) n * (n + 1) * (2L * n + 1) / 6;
    }
}
```

### Example

```
Sum of squares 1..1 brute=1, formula=1
Sum of squares 1..5 brute=55, formula=55
Sum of squares 1..10 brute=385, formula=385
Sum of squares 1..20 brute=2870, formula=2870
```

## Quick Reference Cheat Sheet

| Tool | What it does | Watch out for |
|---|---|---|
| Greedy + Math | justify a greedy choice via an exchange argument (→ Fractional Knapsack) | greedy strategies don't transfer between problems automatically |
| Observation | spot a pattern (often periodic) that shortcuts a loop (→ Nim Game) | always confirm against brute force on small cases first |
| Mathematical Proof | prove a formula/algorithm correct, usually via induction | a formula that "looks right" on a few cases isn't proven |
| Formula Derivation | turn a recurrence or sum into a closed form | telescoping sums are a common derivation technique |
