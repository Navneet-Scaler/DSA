# 20. Simulation

Sometimes the fastest path to a correct answer is to **simulate the process exactly as described** — step by step — rather than searching for a clever closed-form shortcut. This lesson covers three flavors: simulating a real-world process, simulating a numeric rule applied repeatedly, and simulating a mathematical process to estimate or verify a result.

## 1. Process Simulation

### Theory

Process simulation directly models a described procedure — often literally running the steps with a data structure that mirrors the real-world setup. **Classic example — the Josephus Problem:** `n` people stand in a circle; starting from person `0`, every `k`-th person is eliminated until one remains. You can simulate this literally with a circular data structure, or use the much faster **recurrence** that skips the simulation entirely:

```
J(1) = 0
J(n) = (J(n-1) + k) mod n     (0-indexed position of the survivor, for n people)
```

### Code

```java
import java.util.LinkedList;

public class ProcessSimulation {
    public static void main(String[] args) {
        int n = 7, k = 3;

        System.out.println("Survivor via direct simulation (0-indexed): " + josephusSimulate(n, k));
        System.out.println("Survivor via recurrence (0-indexed): " + josephusRecurrence(n, k));
    }

    // Literal simulation: O(n*k) using a rotating list
    static int josephusSimulate(int n, int k) {
        LinkedList<Integer> people = new LinkedList<>();
        for (int i = 0; i < n; i++) people.add(i);

        int index = 0;
        while (people.size() > 1) {
            index = (index + k - 1) % people.size();
            people.remove(index);
        }
        return people.get(0);
    }

    // Recurrence: O(n), no simulation needed
    static int josephusRecurrence(int n, int k) {
        int result = 0;
        for (int i = 2; i <= n; i++) {
            result = (result + k) % i;
        }
        return result;
    }
}
```

### Example

```
Survivor via direct simulation (0-indexed): 3
Survivor via recurrence (0-indexed): 3
```

Both methods agree: with 7 people and every 3rd eliminated, position `3` (the 4th person) survives.

### Problem (Advanced): Bulb Switcher

**Theory:** **Classic example — Bulb Switcher:** `n` bulbs start off; on round `i` (from `1` to `n`), toggle every bulb whose position is a multiple of `i`. Count how many bulbs are on after all `n` rounds. Directly simulating this is `O(n log n)` (bulb `j` gets toggled once per divisor it has), but there's a sharp shortcut: bulb `j` ends up **on** exactly when it has an **odd number of divisors** — and a number has an odd divisor count only when it's a **perfect square** (every divisor pairs with a distinct partner except a square root, which pairs with itself — the same fact used for [Count of Divisors](<./04. Divisibility.md#problem-1-basic-count-of-divisors>) in Lesson 4). So the answer is simply the count of perfect squares `≤ n`, which is `⌊√n⌋` — an `O(1)` formula replacing the full simulation entirely.

```java
public class BulbSwitcher {
    public static void main(String[] args) {
        System.out.println("bulbSwitch(3) [simulate]: " + bulbSwitchSimulate(3)
                + ", [formula]: " + bulbSwitchFormula(3));
        System.out.println("bulbSwitch(99) [simulate]: " + bulbSwitchSimulate(99)
                + ", [formula]: " + bulbSwitchFormula(99));
    }

    // Literal simulation: O(n log n)
    static int bulbSwitchSimulate(int n) {
        boolean[] bulbs = new boolean[n + 1];
        for (int i = 1; i <= n; i++) {
            for (int j = i; j <= n; j += i) {
                bulbs[j] = !bulbs[j];
            }
        }
        int count = 0;
        for (int i = 1; i <= n; i++) if (bulbs[i]) count++;
        return count;
    }

    // Closed form: O(1), since only perfect-square positions end up "on"
    static int bulbSwitchFormula(int n) {
        return (int) Math.sqrt(n);
    }
}
```

```
bulbSwitch(3) [simulate]: 1, [formula]: 1
bulbSwitch(99) [simulate]: 9, [formula]: 9
```

`bulbSwitch(99) = 9` — matching `⌊√99⌋ = 9` exactly, since the perfect squares `1, 4, 9, ..., 81` (nine of them) are the only bulb positions left on.

## 2. Number Simulation

### Theory

Some problems define a **numeric rule** applied repeatedly to a value, and ask what happens after many iterations — simulate the rule directly since there's often no shortcut formula. **Classic example — Happy Numbers:** repeatedly replace a number with the sum of the squares of its digits (see [Lesson 3: Digits](<./03. Digits.md>)); the number is "happy" if this process reaches `1`, or "unhappy" if it falls into a repeating cycle that never includes `1`.

### Code

```java
import java.util.HashSet;
import java.util.Set;

public class NumberSimulation {
    public static void main(String[] args) {
        System.out.println("19 is happy: " + isHappy(19));
        System.out.println("4 is happy: " + isHappy(4));
    }

    static boolean isHappy(int n) {
        Set<Integer> seen = new HashSet<>();
        while (n != 1 && !seen.contains(n)) {
            seen.add(n);
            n = sumOfSquaredDigits(n);
        }
        return n == 1;
    }

    static int sumOfSquaredDigits(int n) {
        int total = 0;
        while (n > 0) {
            int digit = n % 10;
            total += digit * digit;
            n /= 10;
        }
        return total;
    }
}
```

### Example

```
19 is happy: true
4 is happy: false
```

`19 → 82 → 68 → 100 → 1` reaches `1` (happy). `4 → 16 → 37 → 58 → 89 → 145 → 42 → 20 → 4` returns to `4`, a cycle that never hits `1` (unhappy).

### Problem (Medium): Robot Bounded in Circle

**Theory:** **Classic example — Robot Bounded in Circle:** a robot on an infinite grid repeatedly executes a fixed instruction string (`G` = move forward, `L`/`R` = turn 90°). Determine whether the robot's path is **bounded** (stays within some circle forever) rather than escaping to infinity. Key insight: running the instructions **once** either returns the robot to the origin, **or** it doesn't — but if it doesn't return **and** ends up facing the same direction it started, then repeating the instructions forever traces the *same* displacement vector over and over, drifting away in a straight line (unbounded). If it ends up facing any **other** direction, the repeated displacement vectors rotate around and must eventually cancel out over at most 4 repetitions — bounded. So the whole infinite simulation reduces to checking just **one pass**.

```java
public class RobotBoundedInCircle {
    public static void main(String[] args) {
        System.out.println("isRobotBounded('GGLLGG'): " + isRobotBounded("GGLLGG"));
        System.out.println("isRobotBounded('GG'): " + isRobotBounded("GG"));
        System.out.println("isRobotBounded('GL'): " + isRobotBounded("GL"));
    }

    static boolean isRobotBounded(String instructions) {
        int x = 0, y = 0;
        int dx = 0, dy = 1; // facing north

        for (char c : instructions.toCharArray()) {
            if (c == 'G') {
                x += dx;
                y += dy;
            } else if (c == 'L') {
                int temp = dx;
                dx = -dy;
                dy = temp;
            } else if (c == 'R') {
                int temp = dx;
                dx = dy;
                dy = -temp;
            }
        }

        // Bounded if back at origin, OR no longer facing the original direction (north)
        return (x == 0 && y == 0) || !(dx == 0 && dy == 1);
    }
}
```

```
isRobotBounded('GGLLGG'): true
isRobotBounded('GG'): false
isRobotBounded('GL'): true
```

`"GG"` moves straight north and ends still facing north — repeating it forever drifts unboundedly northward. `"GL"` ends facing west (a 90° turn) — even though it isn't back at the origin, the changed facing guarantees the path eventually loops back around, bounding it.

## 3. Mathematical Simulation

### Theory

**Monte Carlo simulation** estimates a mathematical quantity by repeated random sampling rather than deriving it analytically. **Classic example — estimating π:** generate random points in the unit square `[0,1) × [0,1)`; the fraction that land inside the quarter unit circle (`x² + y² ≤ 1`) approximates `π/4`, since the quarter circle's area is `π/4` and the square's area is `1`:

```
π ≈ 4 × (points inside the circle) / (total points)
```

Accuracy improves (slowly) as the sample count grows — this is the same probabilistic idea as [Lesson 19: Uniform Distribution](<./19. Probability.md#4-uniform-distribution>), applied to approximate a constant instead of testing randomness itself.

### Code

```java
import java.util.Random;

public class MathematicalSimulation {
    public static void main(String[] args) {
        Random random = new Random(7);
        int totalSamples = 100_000;

        double piEstimate = estimatePi(totalSamples, random);
        System.out.println("Pi estimate (seed 7, " + totalSamples + " samples): " + piEstimate);
    }

    static double estimatePi(int samples, Random random) {
        int insideCircle = 0;
        for (int i = 0; i < samples; i++) {
            double x = random.nextDouble();
            double y = random.nextDouble();
            if (x * x + y * y <= 1.0) {
                insideCircle++;
            }
        }
        return 4.0 * insideCircle / samples;
    }
}
```

### Example

```
Pi estimate (seed 7, 100000 samples): 3.13576
```

Not exact (that's the nature of Monte Carlo methods — error shrinks roughly with `1/√samples`), but it's in the right neighborhood of `π ≈ 3.14159` using nothing but random sampling and a simple geometric test.
