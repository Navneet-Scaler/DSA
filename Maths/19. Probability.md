## 1. Basic Probability

### Theory

The probability of an event is the count of **favorable outcomes** divided by the count of **total possible outcomes** (assuming all outcomes are equally likely):

```
P(event) = favorable outcomes / total outcomes
```

**Classic example:** rolling two fair dice, what's the probability the sum is `7`? There are `6 × 6 = 36` total outcomes, and `6` of them sum to `7` (`(1,6),(2,5),(3,4),(4,3),(5,2),(6,1)`).

### Code

```java
public class BasicProbability {
    public static void main(String[] args) {
        int favorable = 0;
        int total = 0;

        for (int die1 = 1; die1 <= 6; die1++) {
            for (int die2 = 1; die2 <= 6; die2++) {
                total++;
                if (die1 + die2 == 7) favorable++;
            }
        }

        double probability = (double) favorable / total;
        System.out.println("P(sum = 7) = " + favorable + "/" + total + " = " + probability);
    }
}
```

### Example

```
P(sum = 7) = 6/36 = 0.16666666666666666
```

### Problem (Advanced): New 21 Game

**Theory:** A player draws random integers from `1` to `maxPts` (uniformly), stopping as soon as their running total is `≥ k`. Find the probability their final total is `≤ n`. Direct simulation over every possible draw sequence is exponential — instead, `dp[i]` = probability of the running total ever landing **exactly** on `i`. Since each of the `maxPts` next values is equally likely, `dp[i]` depends on the **average** of the previous `maxPts` `dp` values (a sliding-window sum, updated in O(1) per step rather than resumming the whole window). Once the total reaches `k` or beyond, drawing stops — so only `dp` values for totals `< k` contribute to future totals, while `dp` values for totals in `[k, n]` are summed directly into the final answer.

```java
public class New21Game {
    public static void main(String[] args) {
        System.out.println("new21Game(10,1,10): " + new21Game(10, 1, 10));
        System.out.println("new21Game(6,1,10): " + new21Game(6, 1, 10));
        System.out.println("new21Game(21,17,10): " + new21Game(21, 17, 10));
    }

    static double new21Game(int n, int k, int maxPts) {
        if (k == 0 || n >= k + maxPts) return 1.0;

        double[] dp = new double[n + 1];
        dp[0] = 1.0;
        double windowSum = 1.0;
        double result = 0.0;

        for (int i = 1; i <= n; i++) {
            dp[i] = windowSum / maxPts;
            if (i < k) {
                windowSum += dp[i];
            } else {
                result += dp[i];
            }
            if (i - maxPts >= 0 && i - maxPts < k) {
                windowSum -= dp[i - maxPts];
            }
        }
        return result;
    }
}
```

```
new21Game(10,1,10): 0.9999999999999999
new21Game(6,1,10): 0.6
new21Game(21,17,10): 0.7327777870686095
```

`new21Game(10,1,10)` is mathematically exactly `1.0` (drawing once, uniformly from `1..10`, always lands `≤ 10`) — the tiny `0.9999999999999999` is ordinary floating-point summation drift, exactly the kind of thing flagged in [Maths Lesson 11: Floating Point Errors](<./11. Decimal Mathematics.md#2-floating-point-errors>).

## 2. Expected Value

### Theory

The **expected value** of a random variable is the probability-weighted average of all its possible outcomes:

```
E[X] = Σ (xᵢ × P(xᵢ))
```

For a fair 6-sided die, every face is equally likely (`P = 1/6`), so `E[X]` is just the plain average of `1..6`. For the **sum of two dice**, weight each possible sum by how many of the 36 outcomes produce it.

### Code

```java
public class ExpectedValue {
    public static void main(String[] args) {
        System.out.println("E[single die]: " + expectedSingleDie());
        System.out.println("E[sum of two dice]: " + expectedTwoDice());
    }

    static double expectedSingleDie() {
        double sum = 0;
        for (int face = 1; face <= 6; face++) {
            sum += face * (1.0 / 6);
        }
        return sum;
    }

    static double expectedTwoDice() {
        double sum = 0;
        int total = 36;
        for (int die1 = 1; die1 <= 6; die1++) {
            for (int die2 = 1; die2 <= 6; die2++) {
                sum += (die1 + die2) * (1.0 / total);
            }
        }
        return sum;
    }
}
```

### Example

```
E[single die]: 3.5
E[sum of two dice]: 7.0
```

## 3. Random Selection

### Theory

Two classic patterns for picking randomly:

1. **Fisher-Yates Shuffle** — produces a uniformly random permutation of an array by walking from the last index to the first, swapping each position with a uniformly random earlier-or-equal position.
2. **Reservoir Sampling** — picks one element **uniformly at random from a stream of unknown length**, seeing each element only once. Keep the first element as the current pick; for each subsequent element `i` (0-indexed), replace the pick with it with probability `1/(i+1)`. This guarantees every element ends up with equal probability `1/n`, even though `n` isn't known in advance.

### Code

```java
import java.util.Random;

public class RandomSelection {
    public static void main(String[] args) {
        Random random = new Random(42);
        int[] arr = { 1, 2, 3, 4, 5 };
        fisherYatesShuffle(arr, random);
        System.out.print("Shuffled [1,2,3,4,5] with seed 42: ");
        printArray(arr);

        Random random2 = new Random(42);
        int[] stream = { 10, 20, 30, 40, 50 };
        System.out.println("Reservoir sample (seed 42): " + reservoirSampleOne(stream, random2));
    }

    // Uniformly random permutation, O(n)
    static void fisherYatesShuffle(int[] arr, Random random) {
        for (int i = arr.length - 1; i > 0; i--) {
            int j = random.nextInt(i + 1);
            int temp = arr[i];
            arr[i] = arr[j];
            arr[j] = temp;
        }
    }

    // Pick one element uniformly at random from a stream, seen one at a time, O(n)
    static int reservoirSampleOne(int[] stream, Random random) {
        int chosen = stream[0];
        for (int i = 1; i < stream.length; i++) {
            int j = random.nextInt(i + 1);
            if (j == 0) {
                chosen = stream[i];
            }
        }
        return chosen;
    }

    static void printArray(int[] arr) {
        StringBuilder sb = new StringBuilder("[");
        for (int i = 0; i < arr.length; i++) {
            sb.append(arr[i]);
            if (i < arr.length - 1) sb.append(", ");
        }
        sb.append("]");
        System.out.println(sb);
    }
}
```

### Example

```
Shuffled [1,2,3,4,5] with seed 42: [2, 3, 4, 5, 1]
Reservoir sample (seed 42): 30
```

> Java's `Random` is deterministic given a seed, so these exact outputs are reproducible — useful for testing randomized code (see the [stress-testing pattern](<./09. Factorials.md>) idea applied elsewhere in these notes).

## 4. Uniform Distribution

### Theory

A **uniform distribution** means every outcome in a range is equally likely — no value is favored over another. `Random.nextInt(bound)` generates integers uniformly in `[0, bound)`; adding an offset shifts the range (e.g. `random.nextInt(6) + 1` simulates a fair die, uniform over `[1, 6]`). Over **many** trials, each value should appear roughly `1/bound` of the time — this is exactly what a large stress test would show as a near-flat frequency histogram.

### Code

```java
import java.util.Random;

public class UniformDistribution {
    public static void main(String[] args) {
        Random random = new Random(1);
        System.out.print("10 simulated die rolls (seed 1): ");
        for (int i = 0; i < 10; i++) {
            System.out.print((random.nextInt(6) + 1) + " ");
        }
        System.out.println();
    }
}
```

### Example

```
10 simulated die rolls (seed 1): 4 5 2 4 3 5 3 5 5 5
```

> With only 10 rolls the distribution won't look perfectly even (`5` shows up often here) — uniformity is a statement about the *long-run* frequency over many trials, not about any single short run.

### Problem (Advanced): Generate Random Point in a Circle

**Theory:** Generate a point **uniformly at random** within a circle of given radius. The naive approach — pick a random radius `r` uniformly in `[0, R]` and a random angle uniformly in `[0, 2π)` — is a classic **gotcha**: it clusters points too densely near the center, because the *area* of a thin ring at radius `r` is proportional to `r` itself, not constant. To sample uniformly by **area**, the radius must instead be `R × √(uniform random in [0,1])` — the square root compensates exactly for the ring-area growth, spreading points evenly across the whole disk instead of bunching them near the middle.

```java
import java.util.Random;

public class RandomPointInCircle {
    private final double radius, centerX, centerY;
    private final Random random;

    public RandomPointInCircle(double radius, double x, double y, Random random) {
        this.radius = radius;
        this.centerX = x;
        this.centerY = y;
        this.random = random;
    }

    public static void main(String[] args) {
        RandomPointInCircle generator = new RandomPointInCircle(5.0, 0.0, 0.0, new Random(42));
        double[] point = generator.randPoint();
        System.out.println("Random point in circle (radius=5, seed=42): ("
                + point[0] + ", " + point[1] + ")");
    }

    public double[] randPoint() {
        double r = radius * Math.sqrt(random.nextDouble()); // sqrt compensates for ring-area growth
        double angle = random.nextDouble() * 2 * Math.PI;
        double x = centerX + r * Math.cos(angle);
        double y = centerY + r * Math.sin(angle);
        return new double[] { x, y };
    }
}
```

```
Random point in circle (radius=5, seed=42): (-1.737366780378331, -3.8949516904911063)
```

The generated point sits at distance `≈4.26` from the center — safely within the radius-`5` circle. Without the `sqrt`, this exact `nextDouble()` sequence would instead scale the radius **linearly**, systematically under-representing points near the outer edge of the circle across many draws.
