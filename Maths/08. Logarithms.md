# 8. Logarithms

Logarithms show up constantly in complexity analysis (binary search is O(log n), balanced trees have O(log n) height) and in practical number problems like counting digits or checking orders of magnitude. This lesson covers Java's logarithm functions and a very practical application: counting digits without a loop.

## 1. log

### Theory

A logarithm answers the question: "what power do I need to raise the **base** to, to get this number?"

```
log_b(x) = y   means   b^y = x
```

`Math.log(x)` in Java computes the **natural logarithm** — logarithm base `e` (Euler's number, ≈ 2.71828). Three algebraic identities make logs useful for simplifying calculations:

| Rule | Identity |
|---|---|
| Product rule | `log(a * b) = log(a) + log(b)` |
| Quotient rule | `log(a / b) = log(a) - log(b)` |
| Power rule | `log(a^n) = n * log(a)` |
| Change of base | `log_b(x) = log(x) / log(b)` (any base, via natural log) |

The change-of-base formula is important because Java only ships `Math.log` (base `e`) and `Math.log10` (base 10) directly — every other base is computed by dividing.

### Code

```java
public class Log {
    public static void main(String[] args) {
        System.out.println("Math.log(100) [natural log]: " + Math.log(100));

        // Product rule check
        System.out.println("log(6):          " + Math.log(6));
        System.out.println("log(2) + log(3): " + (Math.log(2) + Math.log(3)));
    }
}
```

### Example

```
Math.log(100) [natural log]: 4.605170185988092
log(6):          1.791759469228055
log(2) + log(3): 1.791759469228055
```

## 2. log10

### Theory

`Math.log10(x)` computes the base-10 logarithm directly — how many times you'd multiply `10` by itself to reach `x`. It's the natural tool for anything tied to decimal digits: order of magnitude, digit counting, and scientific notation.

> **Java-specific guarantee:** per the `Math.log10` documentation, if the argument is exactly `10^n` for an integer `n`, the result is **exactly `n`** — no floating-point rounding error. This special case is exactly why `Math.log10`-based digit counting (below) is safe to use in Java.

### Code

```java
public class Log10 {
    public static void main(String[] args) {
        System.out.println("Math.log10(100):  " + Math.log10(100));
        System.out.println("Math.log10(1000): " + Math.log10(1000));
        System.out.println("Math.log10(4021): " + Math.log10(4021));
    }
}
```

### Example

```
Math.log10(100):  2.0
Math.log10(1000): 3.0
Math.log10(4021): 3.604334073102911
```

## 3. log2

### Theory

Java has **no built-in `Math.log2`** — you compute it via the change-of-base formula: `log2(x) = log(x) / log(2)` (or equivalently using `log10`). `log2` is especially relevant in DSA because it directly measures binary tree height, binary search steps, and bit-length.

> **Precision caution:** unlike `Math.log10`, there is no documented exact-result guarantee for change-of-base division, since it involves dividing two irrational logarithms. For most values it lands exactly on the integer for powers of two, but relying on `floor(log2(x))` to test "is `x` an exact power of 2?" is fragile. When you need an **exact** answer (not just an estimate), prefer bit tricks instead: `(x & (x - 1)) == 0` checks for an exact power of 2 with zero floating-point risk.

### Code

```java
public class Log2 {
    public static void main(String[] args) {
        double log2Of8 = Math.log(8) / Math.log(2);
        double log2Of1024 = Math.log(1024) / Math.log(2);
        System.out.println("log2(8):    " + log2Of8);
        System.out.println("log2(1024): " + log2Of1024);

        // Safer exact check for "is x a power of 2?" using bit tricks, no floating point involved
        System.out.println("1024 is a power of 2: " + isPowerOfTwo(1024));
        System.out.println("1000 is a power of 2: " + isPowerOfTwo(1000));
    }

    static boolean isPowerOfTwo(int x) {
        return x > 0 && (x & (x - 1)) == 0;
    }
}
```

### Example

```
log2(8):    3.0
log2(1024): 10.0
1024 is a power of 2: true
1000 is a power of 2: false
```

### Problem 1 (Basic): Next Power of Two

**Theory:** Find the smallest power of 2 that is `≥ n`. `log2(n)` tells you exactly how many times 2 must be multiplied to reach `n`; rounding that **up** (`Math.ceil`) gives the needed exponent even when `n` isn't itself a power of 2, and `1 << exponent` reconstructs the actual value via a bit shift.

```java
public class NextPowerOfTwo {
    public static void main(String[] args) {
        System.out.println("nextPowerOfTwo(5): " + nextPowerOfTwo(5));
        System.out.println("nextPowerOfTwo(16): " + nextPowerOfTwo(16));
        System.out.println("nextPowerOfTwo(17): " + nextPowerOfTwo(17));
    }

    static int nextPowerOfTwo(int n) {
        if (n <= 1) return 1;
        int exponent = (int) Math.ceil(Math.log(n) / Math.log(2));
        return 1 << exponent;
    }
}
```

```
nextPowerOfTwo(5): 8
nextPowerOfTwo(16): 16
nextPowerOfTwo(17): 32
```

`16` is already an exact power of 2, so it maps to itself; `17` needs to round up to the next one, `32`.

### Problem 2 (Advanced): Power of Three

**Theory:** Check whether `n` is an exact power of 3. Unlike powers of 2, there's no simple bitwise trick (3 isn't a power of the machine's base-2 representation) — but a **log-based** check works: `log(n) / log(3)` should land almost exactly on an integer if `n` is a power of 3, checked with a small floating-point tolerance. A more robust **exact** alternative (no floating-point risk at all): since `3^19 = 1162261467` is the **largest** power of 3 that fits in a 32-bit `int`, any power of 3 must evenly divide it (because powers of the same base divide each other), giving a division-based check with zero precision concerns.

```java
public class PowerOfThree {
    static final int MAX_POWER_OF_THREE = 1162261467; // 3^19, the largest power of 3 fitting in a 32-bit int

    public static void main(String[] args) {
        for (int n : new int[] { 27, 45, 1 }) {
            System.out.println("isPowerOfThree(" + n + ") log=" + isPowerOfThreeLog(n)
                    + " maxPowerTrick=" + isPowerOfThreeExact(n));
        }
    }

    // Log-based -- works, but carries floating-point risk for very large n
    static boolean isPowerOfThreeLog(int n) {
        if (n <= 0) return false;
        double x = Math.log10(n) / Math.log10(3);
        return Math.abs(x - Math.round(x)) < 1e-10;
    }

    // Exact -- no floating point at all
    static boolean isPowerOfThreeExact(int n) {
        return n > 0 && MAX_POWER_OF_THREE % n == 0;
    }
}
```

```
isPowerOfThree(27) log=true maxPowerTrick=true
isPowerOfThree(45) log=false maxPowerTrick=false
isPowerOfThree(1) log=true maxPowerTrick=true
```

`27 = 3³` divides evenly into `3^19`, confirming it via the exact method; `45 = 9 × 5` isn't a power of 3 at all, so it fails both checks, including leaving a nonzero remainder against `1162261467`.

## 4. Digit Count Using Logs

### Theory

Since `log10(n)` tells you "how many powers of 10 fit into `n`", the digit count of a positive integer `n` is:

```
digitCount(n) = floor(log10(n)) + 1
```

This is an **O(1)** alternative to the loop-based digit counter from [Lesson 3: Digits](<./03. Digits.md>). Thanks to `Math.log10`'s documented exact behavior at powers of 10, this is safe to use in Java for any `n > 0`. It still can't handle `n = 0` (undefined logarithm) — that case must be special-cased, same as the loop-based version.

### Code

```java
public class DigitCountUsingLogs {
    public static void main(String[] args) {
        System.out.println("Digit count of 4021 (log10):  " + digitCount(4021));
        System.out.println("Digit count of 100 (log10):   " + digitCount(100));
        System.out.println("Digit count of 999 (log10):   " + digitCount(999));
        System.out.println("Digit count of 1000 (log10):  " + digitCount(1000));
    }

    static int digitCount(int n) {
        if (n == 0) return 1; // log10(0) is undefined; zero has exactly one digit
        return (int) Math.log10(n) + 1;
    }
}
```

### Example

```
Digit count of 4021 (log10):  4
Digit count of 100 (log10):   3
Digit count of 999 (log10):   3
Digit count of 1000 (log10):  4
```

### Problem (Advanced): Count Digits of Factorial

**Theory:** Find how many digits `n!` has, **without** computing the (potentially astronomically large) factorial itself. The [product rule](#1-log) says `log10(n!) = log10(1) + log10(2) + ... + log10(n)` — summing logs turns the product into a sum, so accumulate `log10(i)` for each `i` from `2` to `n`, then apply the same `floor(...) + 1` digit-count formula from above. This computes the digit count in **O(n)** without ever materializing a giant number (a genuine `BigInteger` factorial for large `n` would take far more time and memory).

```java
public class CountDigitsOfFactorial {
    public static void main(String[] args) {
        System.out.println("countDigitsOfFactorial(10): " + countDigitsOfFactorial(10));
        System.out.println("countDigitsOfFactorial(100): " + countDigitsOfFactorial(100));
        System.out.println("countDigitsOfFactorial(1): " + countDigitsOfFactorial(1));
    }

    static long countDigitsOfFactorial(int n) {
        if (n <= 1) return 1;
        double logSum = 0;
        for (int i = 2; i <= n; i++) {
            logSum += Math.log10(i);
        }
        return (long) logSum + 1;
    }
}
```

```
countDigitsOfFactorial(10): 7
countDigitsOfFactorial(100): 158
countDigitsOfFactorial(1): 1
```

`10! = 3628800` has `7` digits — confirmed both by the log-sum formula and by direct counting. `100!` has `158` digits despite being a number with hundreds of digits worth of magnitude — the log-sum approach finds this instantly, while actually computing `100!` (a ~158-digit `BigInteger`) and counting its digits would be far more expensive.

## Quick Reference Cheat Sheet

| Task | Java expression |
|---|---|
| Natural log (base e) | `Math.log(x)` |
| Base-10 log | `Math.log10(x)` |
| Base-2 log | `Math.log(x) / Math.log(2)` (no built-in) (→ Next Power of Two) |
| Any base `b` | `Math.log(x) / Math.log(b)` (→ Power of Three) |
| Digit count (fast) | `(int) Math.log10(n) + 1`, with `n == 0` special-cased (→ Count Digits of Factorial) |
| Exact power-of-2 check | `(x & (x - 1)) == 0`, prefer over `log2` for exactness |
