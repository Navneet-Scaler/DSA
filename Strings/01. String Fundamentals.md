## 1. What Is a String?

### Theory

A string is a sequence of characters treated as a single value. In Java, `String` is a **class**, not a primitive — internally, it wraps a character array (a `byte[]` since Java 9's "Compact Strings" optimization, using Latin-1 encoding when possible and UTF-16 otherwise) along with cached metadata like the hash code. Every string operation you'll see in this series is ultimately built from array operations already covered in the [Arrays series](<../Arrays/README.md>) — the difference is the extra guarantees (immutability, pooling) Java layers on top.

## 2. Character Encoding (ASCII, Unicode, UTF-8)

### Theory

- **ASCII** — 7-bit encoding covering 128 characters (English letters, digits, basic punctuation). Simple but can't represent accented characters, other scripts, or emoji.
- **Unicode** — a universal standard assigning a unique **code point** to every character across every writing system (over 1.1 million possible code points). Unicode itself is just the *numbering*; it needs an encoding to become bytes.
- **UTF-8** — the dominant Unicode **encoding**: variable-width, 1-4 bytes per character. ASCII characters take exactly 1 byte (fully backward compatible with ASCII), while accented/non-Latin characters take more.

> **Java-specific note:** Java `char` and `String` store text internally as **UTF-16** (each `char` is normally 2 bytes, though "Compact Strings" since Java 9 uses 1 byte per char when the whole string is Latin-1-representable). This is different from UTF-8, which matters the moment you serialize a string to bytes (network, files) — always specify the encoding explicitly (`getBytes(StandardCharsets.UTF_8)`) rather than relying on the platform default.

### Code

```java
import java.nio.charset.StandardCharsets;
import java.util.Arrays;

public class CharacterEncoding {
    public static void main(String[] args) {
        String s = "café";
        System.out.println("Character length: " + s.length());

        byte[] utf8Bytes = s.getBytes(StandardCharsets.UTF_8);
        System.out.println("UTF-8 byte length: " + utf8Bytes.length);
        System.out.println("UTF-8 bytes: " + Arrays.toString(utf8Bytes));
    }
}
```

### Example

```
Character length: 4
UTF-8 byte length: 5
UTF-8 bytes: [99, 97, 102, -61, -87]
```

`"café"` has 4 characters but 5 UTF-8 bytes — the accented `é` alone takes 2 bytes (`-61, -87`, i.e. `0xC3 0xA9`), while the plain ASCII letters take 1 byte each. (Java's `byte` is signed, so bytes ≥ 128 print as negative numbers — `0xC3` = `195` unsigned = `-61` signed.)

## 3. Mutable vs Immutable Strings

### Theory

`String` in Java is **immutable** — once created, its contents can never change. Every "modifying" method (`concat`, `replace`, `substring`, ...) returns a **new** `String` object, leaving the original untouched. This is a deliberate design choice: immutability makes strings safe to share across threads without synchronization, safe to use as `HashMap` keys (the hash code can be cached, since it never changes), and safe to pool (see [next section](#4-java-string-pool-memory-model--interning)) — none of which would be safe if strings could be mutated after creation.

### Code

```java
public class MutableVsImmutable {
    public static void main(String[] args) {
        String original = "hello";
        String modified = original.concat(" world");

        System.out.println("Original (unchanged): " + original);
        System.out.println("Modified (new object): " + modified);
        System.out.println("Same reference? " + (original == modified));
    }
}
```

### Example

```
Original (unchanged): hello
Modified (new object): hello world
Same reference? false
```

## 4. Java String Pool, Memory Model & Interning

### Theory

Because strings are immutable, Java can safely **share** identical string literals instead of allocating a new object every time one appears — this shared storage area is the **String Pool** (part of the heap since Java 7, previously in PermGen). String literals (`"hello"`) are automatically pooled at compile time; strings created with `new String(...)` are **not** pooled by default, even if their content is identical to a pooled literal.

**Interning** (`.intern()`) manually adds a string to the pool (or returns the existing pooled reference if already present) — useful for deduplicating many equal-but-separately-constructed strings to save memory.

### Code

```java
public class StringPoolInterning {
    public static void main(String[] args) {
        String a = "hello";           // pooled literal
        String b = "hello";           // same pooled reference
        String c = new String("hello"); // new object, NOT pooled
        String d = c.intern();        // now points to the pooled reference

        System.out.println("a == b (both literals): " + (a == b));
        System.out.println("a == c (literal vs new String): " + (a == c));
        System.out.println("a == d (literal vs interned): " + (a == d));
        System.out.println("a.equals(c) (content check): " + a.equals(c));
    }
}
```

### Example

```
a == b (both literals): true
a == c (literal vs new String): false
a == d (literal vs interned): true
a.equals(c) (content check): true
```

> **Rule of thumb:** always use `.equals()` to compare string *content*; `==` only tells you whether two references point to the *same object*, which depends on pooling details you shouldn't have to reason about at every comparison site.

## 5. String Length & Character Access

### Theory

`length()` returns the number of `char` units (UTF-16 code units — not necessarily the same as "characters" for text outside the Basic Multilingual Plane, but equivalent for virtually all everyday text). `charAt(index)` retrieves a single character, **0-indexed**, same convention as arrays (see [Arrays Lesson 1](<../Arrays/01. Array Fundamentals.md#4-indexing--0-based-indexing>)) — `charAt(length())` and beyond throws `StringIndexOutOfBoundsException`.

### Code

```java
public class LengthAndAccess {
    public static void main(String[] args) {
        String s = "hello";
        System.out.println("Length: " + s.length());
        System.out.println("First char: " + s.charAt(0));
        System.out.println("Last char: " + s.charAt(s.length() - 1));

        try {
            s.charAt(s.length());
        } catch (StringIndexOutOfBoundsException e) {
            System.out.println("charAt(length()) threw: " + e.getMessage());
        }
    }
}
```

### Example

```
Length: 5
First char: h
Last char: o
charAt(length()) threw: String index out of range: 5
```

## 6. String Comparison & Lexicographical Ordering

### Theory

`equals()` checks content equality (see [String Pool](#4-java-string-pool-memory-model--interning) for why `==` is unreliable). `compareTo()` returns an `int` describing **lexicographical order** — comparing character by character (by Unicode value) until a difference is found, returning that difference; if one string is a **prefix** of the other, the result is the **length difference** instead.

```
"apple".compareTo("banana")  -> 'a' - 'b' = -1        (negative: "apple" comes first)
"apple".compareTo("app")     -> 5 - 3 = 2              (positive: "apple" is longer, same prefix)
```

### Code

```java
public class ComparisonAndOrdering {
    public static void main(String[] args) {
        System.out.println("'apple'.compareTo('banana'): " + "apple".compareTo("banana"));
        System.out.println("'apple'.compareTo('app'): " + "apple".compareTo("app"));
        System.out.println("'apple'.equals('Apple'): " + "apple".equals("Apple"));
        System.out.println("'apple'.equalsIgnoreCase('Apple'): " + "apple".equalsIgnoreCase("Apple"));
    }
}
```

### Example

```
'apple'.compareTo('banana'): -1
'apple'.compareTo('app'): 2
'apple'.equals('Apple'): false
'apple'.equalsIgnoreCase('Apple'): true
```

## 7. Time Complexity of String Operations

### Theory

| Operation | Complexity | Why |
|---|---|---|
| `charAt(i)`, `length()` | O(1) | direct array access / cached field |
| `substring()` (Java 7+) | O(n) | copies the relevant character range into a new array |
| `concat()` / `+` | O(n) | allocates and copies into a new, larger array |
| `equals()` | O(n) worst case | must compare character by character until a mismatch |
| `compareTo()` | O(n) worst case | same reasoning as `equals()` |
| `indexOf()` (naive) | O(n×m) worst case | see [Lesson 9: Pattern Matching](<./09. Pattern Matching.md>) for faster alternatives |

> **The classic trap:** concatenating in a loop with `+` is **O(n²)** overall — each `+` allocates a new string and copies everything so far, so building an `n`-character string one piece at a time does O(1+2+3+...+n) = O(n²) work. [StringBuilder](#8-string-vs-stringbuilder-vs-stringbuffer) fixes this.

## 8. String vs StringBuilder vs StringBuffer

### Theory

| | `String` | `StringBuilder` | `StringBuffer` |
|---|---|---|---|
| Mutable? | No | Yes | Yes |
| Thread-safe? | Yes (immutable, so trivially safe) | No | Yes (synchronized methods) |
| Performance | Slow for repeated modification | Fast | Slower than `StringBuilder` (synchronization overhead) |
| When to use | Fixed text, map keys, sharing across threads | Building/modifying strings in a single thread | Building/modifying strings shared across threads |

In practice: default to `StringBuilder` for any loop that builds up a string — `StringBuffer` is a legacy holdover from before Java's memory model was well understood, and true multi-threaded string building is rare enough that reaching for `StringBuilder` plus your own synchronization (or better, avoiding shared mutable state entirely) is usually the better call.

### Code

```java
public class StringBuilderVsBuffer {
    public static void main(String[] args) {
        // The O(n^2) trap
        String result = "";
        for (int i = 0; i < 5; i++) {
            result = result + i; // allocates a new String every iteration
        }
        System.out.println("Built with +: " + result);

        // The O(n) fix
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < 5; i++) {
            sb.append(i); // mutates the same underlying buffer
        }
        System.out.println("Built with StringBuilder: " + sb.toString());
    }
}
```

### Example

```
Built with +: 01234
Built with StringBuilder: 01234
```

Same output, but the `StringBuilder` version does O(n) total work across the loop instead of O(n²) — the difference becomes dramatic once `n` is in the thousands or more (full coverage of the construction patterns is in [Lesson 15](<./15. String Construction.md>)).

## Quick Reference Cheat Sheet

| Concept | Key fact |
|---|---|
| String | immutable, backed internally by a byte/char array + cached hash |
| Character Encoding | ASCII ⊂ Unicode; UTF-8 is variable-width (1-4 bytes); Java strings are UTF-16 internally |
| Mutable vs Immutable | `String` is immutable; every "modification" returns a new object |
| String Pool / Interning | literals are pooled automatically; `new String()` isn't; `.intern()` forces pooling |
| Length & Access | `length()` and `charAt(i)` are O(1), 0-indexed |
| Comparison | `.equals()` for content, `==` for reference identity, `.compareTo()` for lexicographic order |
| Time Complexity | most operations are O(n); `+` in a loop is the O(n²) trap |
| String vs StringBuilder vs StringBuffer | immutable/slow vs mutable+fast vs mutable+thread-safe |
